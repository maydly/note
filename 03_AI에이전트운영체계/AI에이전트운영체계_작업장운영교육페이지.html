작업장 AI 에이전트 운영 체계 정리

VS Code 작업장에서 AI 에이전트를 어떻게 세팅하고, 어떤 규정을 학습시키며, 프로젝트별로 어떤 역할과 프로세스로 활용하고 있는지 정리하는 기준 페이지입니다. 사용자가 아닌 사람이 보더라도 구조와 의도를 이해할 수 있도록 풀어쓴 버전입니다.

범위: 작업장 전체 Claude 에이전트: 9개 Codex 설정: 5개 버전: 260619_1730_v3.0 게시 위치: bucket0825.github.io/note
root작업장 루트에서 00_작업장_위치맵 확인
rulesCLAUDE.md와 프로젝트별 규정 분리

gate조사와 계획은 자동, 파일 수정은 승인 후 진행
record결과물은 버전, 보고서, 기록으로 남김

1. 이 문서가 정리하려는 것

목적은 특정 사람에게 간단히 사용법을 알려주는 것이 아니라, 현재 작업장이 AI를 어떻게 조직화해서 쓰고 있는지 설명하는 것입니다.

핵심 목적

이 페이지는 VS Code에서 AI 에이전트를 불러 쓰는 방법만 설명하는 문서가 아닙니다. 더 큰 목적은 사용자가 작업장 전체를 어떤 기준으로 나누고, 각 프로젝트에 어떤 규정을 심어두었고, 여러 AI 에이전트를 어떤 역할로 배정해 업무 흐름을 만들었는지 정리하는 것입니다.

그래서 이 문서는 “AI를 쓰는 법”보다 “AI가 실수하지 않게 일하는 판을 어떻게 만들었는가”를 설명합니다. 제3자가 읽어도 이해할 수 있도록 풀어쓰지만, 주 목적은 외부 전달이 아니라 현재 운영 체계를 기록하고 교육 가능한 형태로 만드는 것입니다.

현재 문서에 포함하는 범위

이 문서는 현재 운영 기준을 설명하는 문서입니다. 따라서 작업장 루트의 공통 규정, 현재 활성 에이전트, 현재 Codex 설정, 현재 명령어, 현재 project-rules, 그리고 주요 프로젝트별 CLAUDE.md를 중심으로 정리합니다.

반대로 _archive 안의 구버전 설정, 폐기보존 파일, 과거 백업 전체는 현재 운영 기준이 아니므로 본문에서는 역사적 참고로만 다룹니다. 정말 “작업장에 존재하는 모든 에이전트 관련 파일의 전수 목록”이 필요하다면 별도 부록이나 인벤토리 페이지로 분리하는 것이 맞습니다.

정리 범위

작업장 전체

루트 작업장, 공통 규정, 프로젝트 지도, 에이전트 폴더, Codex 설정, 각 프로젝트의 CLAUDE.md까지 포함합니다.

설명 방식

요약보다 풀이 우선

각 에이전트가 무슨 일을 하는지뿐 아니라, 왜 그 역할이 필요해졌는지와 어떤 사고를 막는지까지 설명합니다.

게시 전 기준

공개 가능한 형태로 정리

이 공개본은 https://bucket0825.github.io/note/에 올리기 위한 버전입니다. 내부 경로와 민감 정보는 설명형 문장으로 정리하고, 공개에 불필요한 환경 세부는 줄였습니다.

2. 작업장 구조부터 이해하기

이 작업장의 핵심은 “모든 일을 한 폴더에 섞지 않고, 프로젝트별 판단 기준을 분리한다”는 데 있습니다.

작업장 루트는 여러 프로젝트를 담는 상위 공간입니다. 여기에는 작업장 공통 규칙인 CLAUDE.md, 위치를 알려주는 00_작업장_위치맵.md, Claude 에이전트가 들어 있는 .claude/agents, Codex용 에이전트 설정이 들어 있는 .codex/agents가 있습니다.

중요한 점은 AI가 파일을 열 때마다 “지금 어떤 프로젝트의 일을 하고 있는지”를 먼저 식별해야 한다는 것입니다. 같은 사용자가 요청하더라도 maydly, NYC Studio, PA, 수빈 프로젝트는 목적과 말투와 책임 범위가 다릅니다. 이 구분을 하지 않으면 AI는 한 프로젝트의 기준을 다른 프로젝트에 가져와서 그럴듯하지만 위험한 결과물을 만들 수 있습니다.

위치 역할 왜 중요한가
CLAUDE.md 작업장 공통 지침 요청을 받았을 때 에이전트가 어떤 순서로 움직이고, 언제 멈추고, 어떤 기준으로 프로젝트를 분리해야 하는지 정합니다.
00_작업장_위치맵.md 작업장 지도 새 업무를 시작할 때 어느 프로젝트와 업무ID에 속하는지 확인하는 첫 기준입니다. 파일이 많아질수록 이 지도가 중요해집니다.
.claude/agents Claude 에이전트 9개 계획, 구현, 검수, 배포 점검, 기록처럼 업무 단계를 역할별로 나누어 AI가 한 번에 모든 권한을 갖지 않게 합니다.
.codex/agents Codex용 설정 5개 Codex 환경에서도 비슷한 역할 기준으로 계획, 구현, 검수, 릴리즈, 문구 검토를 수행할 수 있게 해둔 설정입니다.
docs/ 작업장 공통 업무 문서 프로젝트 하나에 속하지 않는 에이전트, 보안, 작업장 운영 같은 공통 업무를 업무ID 기준으로 보관합니다.

3. 왜 에이전트를 여러 명으로 나눴나

한 AI가 모든 일을 바로 처리하면 빠르지만, 방향 착오와 파일 사고가 커질 수 있습니다.

문제

AI는 맥락을 섞기 쉽다

같은 사람이 여러 프로젝트를 맡고 있으면 AI는 “이 사람의 스타일”을 하나로 묶어버리기 쉽습니다. 하지만 이 작업장에서는 maydly의 브랜드 오너로서의 말투와 NYC Studio의 운영 정리자 역할이 완전히 다릅니다. 에이전트 규정은 이 경계를 계속 확인하게 만드는 안전장치입니다.

문제

AI는 바로 만들려고 한다

사용자가 “만들어줘”라고 하면 AI는 곧장 파일을 생성하거나 수정하려고 합니다. 그런데 이 작업장은 기존 파일 보존, 버전 분리, 배포 영향 확인이 중요합니다. 그래서 계획과 구현 사이에 승인 게이트를 둬서, 실제 파일을 바꾸기 전에 한 번 멈추게 했습니다.

해결

역할을 나누면 책임이 선명해진다

plan-agent는 계획만 하고, build-agent는 승인된 범위만 구현하고, qa-agent는 직접 고치지 않고 문제를 찾습니다. 이렇게 나누면 “누가 판단했고, 누가 만들었고, 누가 검수했는지”가 기록에 남습니다.

해결

기록이 남아 다음 작업을 이어갈 수 있다

여러 기기와 여러 세션에서 작업하면 대화 맥락이 사라질 수 있습니다. scribe-agent와 report-agent는 작업이 어디까지 진행됐는지, 어떤 결정을 했는지, 어떤 파일을 만들었는지 남기기 위해 존재합니다.

4. 에이전트 체계가 만들어진 흐름

현재 구조는 처음부터 완성형으로 생긴 것이 아니라, 작업 중 생긴 문제를 막기 위해 단계적으로 확장된 체계입니다.

초기에는 plan, build, qa, release처럼 일반적인 개발 흐름에 가까운 역할 분리가 중심이었습니다. 하지만 실제 작업을 하다 보니 “계획은 잘 세웠지만 방향 자체를 먼저 의심해야 하는 상황”, “결과물은 만들었지만 나중에 찾기 어려운 상황”, “Claude가 만든 결과를 다른 모델 관점에서 다시 봐야 하는 상황”이 생겼습니다. 그래서 에이전트 체계가 단순 실행 흐름에서 운영 체계로 확장되었습니다.

기존 보고서 기준으로 2026-06-02에는 Claude Code 에이전트 7개와 GPT Codex 교차감사 1개, 총 8개 체계가 확인되어 있었습니다. 이후 challenge-agent가 전면 흐름에 들어오면서 현재 운영 기준은 9개 에이전트 체계로 보는 것이 맞습니다. 이 변화의 핵심은 “더 많은 AI를 쓰자”가 아니라, AI가 바로 실행하지 않고 생각하고 멈추고 검수하고 기록하게 만드는 것입니다.

시기/단계 구성 변화 왜 필요했나
초기 기본 흐름 plan → 승인 → build → qa → release 요청을 바로 구현하지 않고, 계획과 검수를 거치게 하기 위한 기본 안전장치였습니다.
보고 체계 추가 report-agent 추가 작업 결과가 채팅에만 흩어지면 나중에 추적이 어렵기 때문에, HTML 통합 보고서로 남기는 역할이 필요했습니다.
교차감사 추가 codex-audit-agent 추가 Claude가 만든 결과를 같은 관점으로만 검수하지 않고, Codex를 통해 독립적으로 다시 보기 위한 장치입니다.
방향 검토 추가 challenge-agent 추가 큰 구조 변경, 돈, 계약, 관계, 프로젝트 경계처럼 실행 전에 전제를 점검해야 하는 요청을 막기 위해 생겼습니다.
현재 운영 기준 Claude 에이전트 9개 + Codex 설정 5개 + 공통 명령어 6개 작업장 전체에서 같은 원칙으로 계획, 구현, 검수, 기록, 공개 전 확인을 반복하기 위한 체계입니다.

이력까지 적는 이유

교육용 문서에서 이력이 중요한 이유는, 현재 규칙이 임의로 복잡해진 것이 아니라 실제 문제를 해결하기 위해 생겼다는 것을 보여주기 때문입니다. 에이전트가 9개라는 숫자만 보면 과해 보일 수 있지만, 각각은 “방향을 검토한다”, “계획을 세운다”, “실제로 만든다”, “검수한다”, “다른 모델로 감사한다”, “공개 전에 확인한다”, “보고서로 남긴다”, “기록으로 남긴다”처럼 서로 다른 사고 방지 역할을 갖고 있습니다.

5. Claude 에이전트 9개 상세 설명

현재 작업장 루트의 .claude/agents 기준으로 9개 에이전트가 구성되어 있습니다.

에이전트 하는 일 왜 만들었나
challenge-agent 큰 방향, 구조 변경, 장기 운영, 비용, 계약, 프로젝트 경계 침범 가능성을 먼저 검토합니다. 사용자의 지시를 바로 실행하면 빠르지만, 전제가 틀렸을 때 큰 손실이 납니다. 이 에이전트는 실행 전에 “이 방향이 맞는지”를 한번 반대편에서 검토하기 위해 만들어졌습니다.
plan-agent 요청을 항목별로 분해하고, 관련 파일과 구조를 조사한 뒤 작업 계획을 세웁니다. AI가 바로 코드를 고치거나 파일을 만들지 않게 하기 위한 계획 담당입니다. 계획에는 수정 대상, 리스크, 테스트 방법, 에이전트 배정이 포함됩니다.
brand-text-agent 문구, 브랜드 톤, 안내 흐름, 고객에게 보이는 표현이 프로젝트 기준에 맞는지 검토합니다. 문구를 예쁘게 만드는 것보다 중요한 것은 역할과 기준을 흐리지 않는 것입니다. maydly, NYC, PA, 수빈은 톤이 다르기 때문에 문구 전용 검토자가 필요합니다.
build-agent 승인된 계획이나 명시된 범위 안에서 실제 파일 생성과 수정을 담당합니다. 구현 권한이 있는 에이전트를 따로 분리해, 승인되지 않은 변경을 막기 위해 만들었습니다. 기존 구조 재사용, 덮어쓰기 금지, 새 버전 생성이 핵심 규칙입니다.
qa-agent 구현 결과가 요청과 계획에 맞는지 검수하고, 누락이나 기존 기능 훼손을 찾습니다. 만든 사람이 바로 통과시키면 놓치는 부분이 생깁니다. qa-agent는 직접 수정하지 않고 문제를 찾아 보고하는 검수 전용 역할입니다.
codex-audit-agent Claude가 만든 결과물을 Codex 관점에서 독립 교차감사합니다. 같은 모델 안에서만 검수하면 시야가 겹칠 수 있습니다. 다른 모델의 관점으로 한 번 더 읽게 하여 누락, 위험, 구조 문제를 찾기 위해 만들어졌습니다.
release-agent 배포나 공유 전에 링크, 이미지, 공개/비공개 데이터, 모바일 화면, 파일 경로를 점검합니다. 파일이 잘 만들어졌어도 공개하면 안 되는 정보가 들어 있거나 경로가 깨질 수 있습니다. 특히 Netlify, Cloudflare Pages, GitHub Pages 게시 전 안전 확인을 위해 필요합니다.
report-agent 작업 사이클의 원 요청, 계획, 구현, QA, 감사, 릴리즈 결과를 HTML 보고서로 종합합니다. 작업이 끝난 뒤 채팅에만 결과가 남으면 나중에 찾기 어렵습니다. 브라우저에서 열어볼 수 있는 보고서로 남겨, 검토와 전달을 쉽게 하기 위해 만들어졌습니다.
scribe-agent 대화, 결정사항, 작업 흐름, 중단 지점, 다음 작업을 마크다운 기록으로 남깁니다. 맥북, 데스크탑, 여러 세션을 오가면 맥락이 끊깁니다. scribe-agent는 작업 기억을 파일로 남겨 다음 사람이 바로 이어받을 수 있게 하는 기록 담당입니다.

권한 차이로 보는 에이전트 구조

에이전트 체계를 이해할 때는 이름보다 권한을 먼저 봐야 합니다. 모든 에이전트가 파일을 고칠 수 있는 것이 아니며, 오히려 대부분은 읽기와 검토에 집중합니다. 이 구조는 “생각하는 역할”과 “실제로 바꾸는 역할”을 분리해 파일 사고를 줄이기 위한 것입니다.

권한 범위 해당 에이전트 의미
읽기·검토 중심 challenge-agent, plan-agent, brand-text-agent, qa-agent, release-agent 파일을 직접 고치기보다 방향, 계획, 문구, 검수, 배포 가능성을 판단합니다. 잘못된 수정이 생기지 않도록 먼저 읽고 멈추는 역할입니다.
구현 가능 build-agent 승인된 계획 안에서 실제 파일을 생성하거나 수정할 수 있습니다. 그래서 가장 강한 권한을 갖지만, 동시에 가장 많은 금지 규칙을 가집니다.
보고서 생성 가능 report-agent 원본 코드를 수정하는 역할이 아니라, 작업 결과를 HTML 보고서로 만드는 역할입니다. Write 권한은 보고서 생성 용도입니다.
기록 생성 가능 scribe-agent 대화와 결정사항을 기록 파일로 남깁니다. 구현을 주도하지 않고, 다음 작업자가 이어받을 수 있게 맥락을 보존합니다.
독립 감사 codex-audit-agent 직접 수정하지 않고 Codex 관점으로 결과물을 다시 봅니다. 감사 대상 내용을 읽고 문제 가능성을 보고하는 역할입니다.

작업 성격별 자동 배정 기준

사용자는 매번 “어떤 에이전트를 써줘”라고 지정하지 않아도 됩니다. 작업장 공통 규정은 요청의 성격을 보고 자동으로 어떤 에이전트가 필요한지 판단하도록 설계되어 있습니다. 다만 자동 배정은 파일을 바로 수정한다는 뜻이 아니라, 조사와 계획을 자동화한다는 뜻입니다.

작업 성격 배정 흐름 풀이
방향·구조·큰 결정 검토 challenge-agent → plan-agent 구조를 바꾸거나 새 시스템을 만들 때는 바로 실행하지 않고, 먼저 방향과 리스크를 검토한 뒤 계획으로 넘어갑니다.
구조 조사·계획만 필요 plan-agent 현재 파일 구조를 보고 어떤 순서로 해야 하는지 정리합니다. 이 단계에서는 파일을 고치지 않습니다.
문구·브랜드 텍스트 brand-text-agent 말투, 고객 안내, 브랜드 표현, 운영 기준이 프로젝트에 맞는지 봅니다. 예쁜 문장보다 정확한 맥락을 우선합니다.
코드·파일 생성 또는 수정 승인 → build-agent → qa-agent 실제 수정은 승인 후에만 진행합니다. 구현 뒤에는 QA가 요청사항 반영과 기존 기능 훼손 여부를 확인합니다.
독립 교차감사 codex-audit-agent Claude가 놓친 문제를 다른 모델 관점에서 찾기 위한 단계입니다. 직접 수정하지 않고 감사 결과만 보고합니다.
최종 공유·배포 전 release-agent 공개 링크, 이미지 경로, 모바일 화면, 내부 정보 노출 가능성을 확인합니다. 특히 note 게시 전 중요합니다.
작업 결과 정리 report-agent, scribe-agent 보고서는 결과물 검토용이고, 기록은 다음 작업 인수인계용입니다. 둘 다 작업 맥락을 잃지 않기 위해 존재합니다.

에이전트 체계의 핵심

이 구조에서 중요한 것은 “에이전트가 많다”가 아닙니다. 중요한 것은 역할마다 권한을 다르게 둔 것입니다. 어떤 에이전트는 읽고 계획만 하며, 어떤 에이전트는 실제 파일을 만들 수 있고, 어떤 에이전트는 검수만 합니다. 이 분리는 AI가 한 번의 응답에서 계획, 구현, 판단, 검수를 모두 스스로 해버리는 위험을 줄입니다.

6. Codex 설정 5개는 무엇인가

작업장에는 Claude 에이전트와 별도로 .codex/agents 아래 Codex용 설정도 있습니다.

Codex 설정은 Claude 에이전트와 완전히 같은 구성은 아닙니다. 현재 확인된 Codex 설정은 plan-agent, build-agent, qa-agent, release-agent, brand-text-agent 5개입니다. 즉 Codex 쪽에서는 계획, 구현, 검수, 배포 점검, 문구 검토에 해당하는 핵심 역할을 재현해 둔 상태입니다.

이 설정의 의미는 “어떤 AI 도구를 쓰든 같은 운영 원칙을 유지하려는 것”입니다. Claude를 쓰든 Codex를 쓰든, 기존 파일 덮어쓰기 금지, 프로젝트 경계 분리, 파일명 구체화, 조사 결과 HTML 출력 같은 기준을 반복해서 적용하기 위한 장치입니다.

Codex 설정 대응 역할 교육 포인트
plan-agent.toml계획 담당요청을 바로 구현하지 않고 구조 조사와 계획을 먼저 하게 만듭니다.
build-agent.toml구현 담당승인된 범위 안에서만 파일을 생성하거나 수정하게 합니다.
qa-agent.toml검수 담당직접 고치지 않고 누락, 오류, 요청 밖 변경을 찾아 보고하게 합니다.
release-agent.toml공유 전 점검링크, 이미지, 공개 데이터, 모바일 화면을 배포 전에 확인하게 합니다.
brand-text-agent.toml문구 검토프로젝트별 말투와 운영 기준을 섞지 않도록 문장을 검토하게 합니다.

7. 공통 명령어 6개는 무엇인가

에이전트가 역할이라면, 명령어는 그 역할을 실제 작업 흐름에서 호출하는 입구입니다.

작업장에는 .claude/commands 아래 6개의 명령어 문서가 있습니다. 이 명령어들은 사용자가 매번 긴 규칙을 다시 말하지 않아도, 정해진 흐름을 반복할 수 있게 만드는 단축 절차입니다. 예를 들어 /plan은 구현 전에 조사와 계획을 먼저 하게 만들고, /execute는 승인된 계획이 있을 때 build, qa, release-check를 순서대로 이어갑니다.

명령어 역할 쉽게 풀면
/plan 요청을 바로 구현하지 않고 계획합니다. “일단 만들지 말고, 무엇을 어디서 어떻게 바꿀지 먼저 조사해서 계획서로 보여줘”라는 명령입니다.
/build 승인된 plan 또는 사용자가 승인한 작업 계획만 구현합니다. “이제 승인했으니, 정해진 범위 안에서만 실제 파일을 만들어줘”라는 단계입니다.
/qa 구현 결과를 원 요청과 계획 기준으로 검수합니다. “만든 사람이 말한 것만 믿지 말고, 실제 파일 기준으로 요청이 반영됐는지 확인해줘”라는 단계입니다.
/fix QA Report에서 발견된 문제만 수정합니다. “검수에서 나온 문제만 최소 수정하고, 다른 부분은 건드리지 마”라는 안전 수정 단계입니다.
/release-check 최종 배포 또는 공유 전에 파일, 경로, 화면, 데이터, 보안을 점검합니다. “공개하거나 전달하기 전에 깨지는 것과 노출되면 안 되는 정보가 없는지 봐줘”라는 단계입니다.
/execute 승인된 계획 이후 build → qa → fix → release-check를 이어 실행합니다. “플랜은 승인됐으니 정해진 자동 순서로 진행하되, 심각한 문제가 있으면 멈춰서 보고해줘”라는 복합 실행 명령입니다.

명령어를 만들었다는 것의 의미

명령어는 단순한 단축키가 아닙니다. 반복되는 업무 방식을 규칙으로 고정한 것입니다. 사용자가 매번 “기존 파일 덮어쓰지 말고, 계획 먼저 세우고, QA 하고, 공개 전 확인해줘”라고 말하지 않아도, 명령어 문서가 그 흐름을 기억하게 합니다. 이것이 작업장을 개인 대화가 아니라 운영 시스템처럼 쓰기 위한 핵심입니다.

8. project-rules는 모든 에이전트의 공통 교과서

.claude/skills/project-rules.md는 에이전트와 명령어가 반복해서 참고하는 공통 규정 파일입니다.

작업장 공통 CLAUDE.md가 전체 운영 헌장이라면, project-rules.md는 실제 작업할 때 보는 세부 교과서에 가깝습니다. 여기에는 작업 순서, 파일명 규칙, 날짜·시간 표기, 조사 결과 HTML 출력, 코드 작업 규칙, HTML 규칙, 예약 시스템 규칙, 피그마 작업 규칙, 문구 규칙, 보안 규칙, 보고 규칙, 설명 방식 규칙이 들어 있습니다.

규칙 묶음 내용 왜 필요한가
공통 작업 원칙 바로 구현하지 않고 조사 → 계획 → 승인 → 구현 → QA → Release Check → 커밋 확인 순서로 진행합니다. AI가 빠르게 만들다가 기존 구조를 망치거나, 검수 없이 완료라고 말하는 일을 막습니다.
파일 작업 규칙 기존 파일 덮어쓰기 금지, 원본 이미지 직접 수정 금지, 폴더 구조 임의 변경 금지, 삭제 전 승인 원칙을 둡니다. 작업장은 버전과 이력이 중요하므로, 되돌릴 수 없는 변경을 막아야 합니다.
파일명 규칙 파일명은 대상 + 세부내용 + 유형이 드러나야 하며, 정리, 자료, 리스트, 보고서, 플랜 단독 사용을 금지합니다. 파일이 많아질수록 열어보지 않고도 어떤 파일인지 알아야 다음 작업자가 길을 잃지 않습니다.
날짜·시간 표기 버전 파일, 보고서, 기록은 YYMMDD_HHMM 형식을 사용하고, 최종본에는 날짜와 버전을 붙이지 않습니다. 최신본과 이력 파일을 구분하고, 같은 파일의 발전 순서를 추적하기 위해 필요합니다.
조사 결과 출력 제품 리스트, 비교 자료, 추천 정보는 채팅으로 길게 쓰지 않고 HTML 파일로 만들고 브라우저로 엽니다. 긴 조사 결과를 나중에 다시 열어보고 전달할 수 있게 하기 위한 보관 방식입니다.
보안·공개 범위 비밀번호, API 키, 관리자 메모, 연락처, 금액, 내부 일정은 공개 페이지에 노출하지 않습니다. Google Drive, GitHub, GitHub Pages, Netlify, Cloudflare Pages 같은 공개/동기화 환경에서 정보 유출을 막기 위해 필요합니다.
설명 방식 한 줄 요약으로 끝내지 않고 원인, 맥락, 선택지, 판단 포인트를 설명합니다. 사용자가 되묻지 않아도 이해하고 판단할 수 있도록 하기 위한 교육형 커뮤니케이션 규칙입니다.

9. 설정 파일은 무엇을 허용하고 있나

에이전트와 규칙만큼 중요한 것이 실제 도구 권한을 어디까지 열어두었는지입니다.

.claude/settings.json과 .claude/settings.local.json은 Claude Code 환경에서 허용된 작업을 기록하는 설정 파일입니다. 여기서 중요한 점은 권한이 무작정 넓게 열려 있는 것이 아니라, 특정 경로나 특정 도구 호출만 허용하도록 되어 있다는 점입니다.

파일 현재 확인된 허용 내용 교육용 풀이
.claude/settings.json Edit(.claude/agents/**), Write(.claude/agents/**) 에이전트 정의 파일을 수정하거나 새로 쓸 수 있도록 허용한 설정입니다. 에이전트 시스템 자체를 정비하기 위해 필요한 권한입니다.
.claude/settings.local.json 일부 로컬 도구 연동과 환경 설정을 위한 예외 권한 특정 로컬 환경에서 필요한 예외 권한입니다. 공개 문서에서는 세부 명령보다 “로컬 예외 권한이 별도 관리된다”는 구조만 이해하면 됩니다.

설정 파일은 공개 전 특히 주의

설정 파일에는 작업 환경의 권한 구조가 드러납니다. 이 페이지를 note에 올릴 때는 “어떤 종류의 권한이 있다”는 설명은 남길 수 있지만, 실제 로컬 명령어나 민감한 내부 경로는 공개 문서에 그대로 적지 않는 것이 안전합니다.

10. 프로젝트별 규정은 왜 따로 있는가

같은 사용자가 관리해도 프로젝트마다 사용자의 역할과 판단 기준이 다릅니다.

작업장 공통 규칙은 전체 흐름을 정하지만, 실제 문서나 파일을 만들 때는 각 프로젝트의 CLAUDE.md가 중요합니다. 프로젝트별 규정은 AI에게 “이 폴더 안에서 사용자를 어떤 역할로 대해야 하는지” 알려줍니다. 이 차이가 없으면 AI는 모든 프로젝트를 같은 톤, 같은 책임 범위, 같은 운영 방식으로 처리하게 됩니다.

프로젝트 사용자의 역할 AI가 지켜야 할 핵심 기준
작업장 공통 여러 프로젝트의 운영 체계를 설계하고 관리하는 사람 요청이 들어오면 프로젝트를 먼저 식별하고, 에이전트 배정과 승인 게이트를 거쳐 처리합니다.
NYC Studio 대표가 아니라 운영 정리자, 예약 프로세스 정리자, 대표 전달용 자료 제작자 대표가 이해하기 쉬운 운영 문서, 고객 안내, 예약 기준, 전달용 패키징을 우선합니다. maydly 기준은 가져오지 않습니다.
maydly 브랜드 오너, 메인 포토그래퍼, 기획자, 최종 의사결정자 브랜드 성장, 촬영 운영, 고객 경험, 협업 구조, 수익 구조를 기준으로 판단합니다. NYC Studio 운영 기준을 가져오지 않습니다.
PA PA 독립 업무를 정리하고 운영하는 담당자 NYC와 같은 회사 안에 있어도 별도 구조로 보고, NYC 운영 문맥이나 비용 기준을 섞지 않습니다.
수빈 수빈이의 생활 루틴을 돕는 조력자 회사 프로젝트가 아니라 개인 생활 루틴 프로젝트로 봅니다. 부담, 감시, 평가처럼 느껴지지 않게 하고 외부 공개를 피합니다.

프로젝트별 풀이

NYC Studio

운영 정리와 대표 전달이 중심

NYC Studio에서 사용자는 브랜드 오너나 최종 결정권자가 아니라, 대표가 검토하고 사용할 수 있도록 자료를 정리하는 사람으로 정의됩니다. 그래서 문서의 목적은 멋진 브랜드 문장보다 운영 기준을 명확히 하는 것입니다. 예약, 환불, 변경, 추가금, 공간 이용 규칙이 흐려지면 안 되고, 고객 안내는 친절하되 책임 기준을 보호해야 합니다.

이 프로젝트는 Cloudflare Pages 라이브 배포 상태라는 제약도 있으므로, 파일 이동, 삭제, 경로 변경은 특히 조심해야 합니다.

maydly

브랜드 오너의 판단 기준

maydly에서 사용자는 브랜드 오너이자 촬영자, 기획자, 최종 의사결정자입니다. 따라서 AI는 고객에게 무조건 맞춰주는 방향이 아니라, 브랜드 톤, 촬영자의 결정권, 수익 구조, 협업 기준을 보호하는 방향으로 판단해야 합니다.

maydly의 문체는 따뜻하지만 가볍지 않아야 하고, 협업과 무료 작업이 혼동되지 않게 해야 합니다. Netlify 라이브 배포 상태이므로, 공개 중인 링크가 깨질 수 있는 파일 이동도 사전 확인이 필요합니다.

PA

NYC와 같은 회사여도 별도 기준

PA는 NYC와 같은 회사 안에 있을 수 있지만, 작업장 규정에서는 독립 프로젝트로 다룹니다. 같은 회사라는 이유로 NYC의 고객 응대 방식, 비용 구조, 운영 문서 톤을 가져오면 안 됩니다.

이 기준은 AI가 “비슷해 보이는 프로젝트를 하나로 묶는 습관”을 막기 위한 것입니다. PA 작업은 PA의 위치맵과 PA의 업무ID 구조 안에서 판단해야 합니다.

수빈

개인 생활 루틴이므로 공개와 톤이 다르다

수빈 프로젝트는 회사 프로젝트가 아니라 개인 생활 루틴 프로젝트입니다. 여기서 사용자는 관리자가 아니라 조력자이며, AI는 평가, 감시, 성과 압박처럼 느껴지는 표현을 피해야 합니다.

수빈 영역의 기록은 개인 생활 기록이므로 외부 공개가 기본 목적이 아닙니다. note에 올릴 문서에서 수빈 프로젝트를 설명할 때도 개인 기록의 세부 내용은 드러내지 않고, 프로젝트 분리 원칙과 공개 주의 정도만 설명하는 것이 안전합니다.

수빈-maydly 예외 통합

작업장 위치맵에는 수빈-maydly라는 예외 통합 식별명도 있습니다. 이것은 사용자가 명시적으로 허용한 통합 업무에 한해서만 쓰는 예외 구조입니다. 예외 통합이 있다고 해서 평소에 수빈 프로젝트와 maydly 프로젝트를 자유롭게 섞어도 된다는 뜻은 아닙니다. 오히려 예외라는 이름을 붙여 별도로 관리하는 이유는, 기본 원칙이 “섞지 않는다”이기 때문입니다.

프로젝트 분리의 실제 의미

“프로젝트를 분리한다”는 말은 폴더만 나눈다는 뜻이 아닙니다. AI가 말투, 책임 범위, 최종 판단자, 공개 가능성, 파일 보관 방식까지 다르게 적용해야 한다는 뜻입니다. 예를 들어 maydly에서는 사용자가 브랜드 오너이지만, NYC Studio에서는 대표에게 전달할 자료를 정리하는 사람입니다. 이 차이를 놓치면 같은 문장도 완전히 다른 의미가 됩니다.

11. 실제 업무가 흘러가는 순서

현재 규정은 “AI가 알아서 다 해라”가 아니라 “조사와 계획은 자동, 실제 수정은 승인 후” 구조입니다.

01

프로젝트 식별

요청이 들어오면 먼저 작업장 지도와 현재 경로를 보고 이것이 작업장 공통인지, NYC인지, maydly인지, PA인지, 수빈 영역인지 확인합니다. 이 단계가 빠지면 프로젝트 경계 침범이 생깁니다.

02

challenge-agent 개입 여부 판단

구조 변경, 새 시스템, 돈, 계약, 장기 운영, 팀 역할, 프로젝트 경계 문제가 있으면 바로 만들지 않고 먼저 방향을 검토합니다. 단순 문구 수정이나 이미 승인된 작은 수정이면 challenge-agent는 건너뜁니다.

03

plan-agent식 조사와 계획

관련 파일, 기존 구조, 데이터 흐름, 버전 규칙, 리스크를 확인합니다. 여기서 계획이 만들어지고, 어떤 에이전트가 필요한지도 정해집니다.

04

승인 게이트

조사와 계획까지는 자동으로 진행할 수 있지만, 실제 파일 생성이나 수정은 사용자 승인 후에 진행합니다. 이 규칙은 파일 사고를 막는 핵심 장치입니다.

05

build-agent식 구현

승인된 범위 안에서만 파일을 만들거나 수정합니다. 기존 파일은 덮어쓰지 않고 새 버전으로 남기며, 파일명은 대상과 세부내용과 유형이 보이게 작성합니다.

06

QA, Codex 감사, 릴리즈 점검

구현 후에는 qa-agent가 요청 반영 여부와 기능 훼손을 확인합니다. 필요하면 codex-audit-agent가 독립적으로 다시 보고, 공유나 게시 전에는 release-agent가 공개 범위와 경로를 확인합니다.

07

보고와 기록

report-agent는 작업 결과를 HTML 보고서로 모으고, scribe-agent는 작업 맥락과 중단 지점을 기록합니다. 작업이 끝나면 git 커밋 여부도 사용자에게 확인합니다. 자동 커밋은 하지 않습니다.

12. 요청을 쓸 때 꼭 들어가야 하는 정보

AI에게 자세히 말해야 하는 이유는 일을 길게 만들기 위해서가 아니라, 잘못된 프로젝트 기준을 적용하지 않게 하기 위해서입니다.

작업장 전체 기준으로 정리할 일이야.

참고할 규정 또는 파일:

- 공개 예정이면 내부 경로, 계정 정보, 민감 정보 제거
- 실제 파일 생성 또는 수정 전 필요한 경우 승인 게이트 통과
