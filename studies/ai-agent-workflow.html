혼자 일해도
혼자가 아닌
작업 방식

하나의 요청을 받으면 검토 → 계획 → 구현 → 검수 → 교차감사 → 정리로 이어지는, 역할을 나눈 AI 에이전트 작업 흐름. 1인 운영을 팀처럼 굴리는 방법.

01 — Idea

왜 역할을 나누나

하나의 AI에게 다 시키는 대신, 단계마다 다른 역할을 맡긴다.

발상

Why

만드는 사람과 검수하는 사람이 같으면, 자기 실수를 잘 못 본다. 그래서 같은 AI라도 역할을 나눠 단계별로 다른 시선을 갖게 한다. 만드는 단계, 검수하는 단계, 의심하고 반론하는 단계, 정리하는 단계를 분리하면 결과의 신뢰도가 올라간다. 혼자 운영하는 작업을 팀처럼 굴리는 셈이다.

핵심: 구현 직전에 한 번 멈춰 사람의 승인을 받는다. AI가 끝까지 알아서 가지 않고, 방향이 정해진 뒤에만 손을 댄다.

02 — Flow

한 사이클의 흐름

요청 하나가 결과물이 되기까지 거치는 단계.

승인 전 — 준비

Before
  1. 방향이 맞는지 먼저 의심해 본다 (반론·확인 질문).
  2. 요청을 항목별로 쪼갠다.
  3. 관련 파일·구조를 조사한다.
  4. 업무 기록 카드를 만든다.
  5. 멈추고, 사람의 승인을 기다린다.

승인 후 — 실행

After
  1. 구현한다.
  2. 검수한다 (요청·계획과 맞는지).
  3. 독립적으로 교차감사한다 (다른 도구의 시선으로).
  4. 문제가 있으면 고친다.
  5. 공유·배포 전 마지막 점검을 한다.
  6. 결과를 보기 좋게 정리한다.
  7. 기록을 남기고, 확인 후 커밋한다.
단순한 오타·문구 수정까지 이 전체를 강제하지 않는다. 일의 크기에 맞춰 단계를 건너뛴다. 도구가 일을 위한 것이지, 일이 도구를 위한 게 아니다.

03 — Roles

역할 지도

단계마다 어떤 역할을 부르는지.

주요 역할

Roles

Challenge

바로 실행하지 않고 전제·방향·리스크를 먼저 따진다.

Plan

구현 전에 요구사항을 정리하고 계획을 세운다.

Build

승인된 계획대로만 구현한다. 시키지 않은 변경 금지.

QA

요청·계획과 맞는지, 빠진 건 없는지 검수만 한다.

Audit

다른 도구로 독립 교차감사. 직접 고치지 않고 보고만.

Scribe

결정·작업 흐름·다음 시작점을 기록으로 남긴다.

각 역할은 자기 일만 한다. 검수·감사 역할은 직접 고치지 않고 보고만 한다. 그래야 시선이 섞이지 않는다.
