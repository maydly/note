같은 데이터

정적 앱(HTML·JS)인데 폰·데스크톱에서 같은 데이터를 보고 싶을 때. 예약 캘린더를 만들며 익힌 "서버 없는 클라우드 동기화" 패턴 — 데이터를 어디 두고, 비밀은 어떻게 보호하고, 왜 localhost로 테스트해야 하는지 정리했다.

정적 앱 클라우드 동기화 토큰 보호 localhost

이 노트를 읽고 나면

"서버 없이 어떻게 데이터를 맞추지?"의 답.

  • ✓정적 앱이 데이터를 여러 기기에서 맞추는 기본 구조를 안다.
  • ✓데이터 파일을 어디에 두는 게 안전한지 안다.
  • ✓접근 토큰을 클라이언트에 두지 않는 이유를 안다.
  • ✓왜 파일을 더블클릭하면 안 되고 localhost가 필요한지 안다.
한 줄 요약 — 화면(정적 파일)은 무료 호스팅에, 데이터는 비공개 저장소에, 그 저장소에 쓰는 열쇠(토큰)는 호스팅의 환경변수에 둔다. 클라이언트엔 비밀을 두지 않는다.

01 — 문제: 정적인데 기기 간 공유

브라우저 저장만으론 부족하다.

정적 앱은 보통 데이터를 브라우저 안(localStorage)에 저장한다. 그런데 이건 그 기기·그 브라우저에만 남는다. 폰에서 넣은 예약이 데스크톱엔 안 보인다. "여러 기기에서 같은 데이터"를 하려면, 데이터를 기기 밖 공용 장소에 두고 서로 읽고 써야 한다.

02 — 데이터를 어디 두나

화면과 데이터를 분리한다.

화면

공개 호스팅

HTML·JS(화면)는 무료 정적 호스팅에 올린다. 누가 봐도 되는 부분.

데이터

비공개 저장소

예약·개인 데이터는 공개되면 안 되므로, 별도의 비공개 저장소에 데이터 파일로 둔다.

핵심 분리 — "보여주는 것(화면)"과 "지켜야 하는 것(데이터)"을 다른 곳에 둔다. 화면은 공개, 데이터는 비공개.

03 — 토큰은 클라이언트에 두지 않는다

비공개 저장소에 쓰려면 열쇠가 필요한데…

비공개 저장소에 데이터를 쓰려면 접근 토큰(열쇠)이 필요하다. 그런데 이 토큰을 화면 파일(HTML·JS)에 박으면, 브라우저로 누구나 훔쳐볼 수 있다. 그래서 토큰은 호스팅의 환경변수(env)에 두고, 데이터를 쓰는 처리는 화면이 아니라 호스팅 쪽(서버리스 함수)에서 하게 한다.

이건 저장소 보안 노트의 핵심과 같다 — 비밀은 클라이언트에 두지 않는다. 실제 PIN·토큰 값은 이런 공개 노트에도 절대 적지 않는다(패턴만).

04 — 동기화 흐름

읽고 → 쓰고 → 다른 기기가 다시 읽는다.

1. 화면 로드호스팅에서 앱 열림
2. 데이터 읽기비공개 저장소에서
3. 수정예약 추가·변경
4. 저장(쓰기)env 토큰으로 안전하게
5. 다른 기기다시 읽으면 반영
항상 켜져 있어야(always-on) 다른 기기가 언제 열어도 최신을 받는다. 관리자용·고객용 화면을 나눠 두면, 공개 화면엔 민감한 조작을 노출하지 않을 수 있다.

05 — 왜 localhost로 테스트하나

파일을 더블클릭하면 동작이 막힌다.

이런 앱을 파일 더블클릭(file://)으로 열면, 데이터 불러오기·저장 같은 동작이 브라우저 보안 정책에 막혀 제대로 안 된다. 그래서 로컬에서 테스트할 땐 작은 로컬 서버를 띄워 localhost 주소로 연다. 그래야 실제 배포 환경과 비슷하게 동작한다.

# 예: 프로젝트 폴더에서 로컬 서버 띄우기
npx serve .        # 또는 python -m http.server
# → http://localhost:3000 으로 열기
"내 PC에선 더블클릭해도 열리던데?" — 화면은 떠도 불러오기·저장이 조용히 실패한다. 반드시 localhost로 확인.

서버가 없는데 어떻게 여러 기기가 같은 데이터를 보나

정적 앱은 화면(HTML·JS)만 있고 데이터를 담는 서버가 없다. 그래서 기본값은 데이터를 브라우저 안(localStorage)에 저장하는데, 이건 그 기기·그 브라우저에만 남는다. 폰에서 넣은 걸 데스크톱이 못 본다. 여러 기기가 같은 데이터를 보려면, 데이터를 기기 밖 공용 장소에 두고 서로 읽고 써야 한다.

여기서 핵심은 "보여주는 것"과 "지켜야 하는 것"을 분리하는 발상이다. 화면은 누가 봐도 되니 공개 호스팅에 둔다. 하지만 예약·개인 데이터는 공개되면 안 되니 비공개 저장소에 데이터 파일로 둔다. 이렇게 나누면 무료 정적 호스팅만으로도 "서버 있는 앱"처럼 동작하게 만들 수 있다.

남는 문제는 "비공개 저장소에 쓰려면 열쇠(토큰)가 필요한데, 그 열쇠를 어디 두나"다. 화면 파일에 두면 브라우저로 누구나 훔쳐본다. 그래서 열쇠는 호스팅의 환경변수(env)에 숨기고, 실제로 쓰는 처리는 화면이 아니라 호스팅 뒤쪽(서버리스 함수)에서 하게 한다. 클라이언트엔 비밀을 절대 두지 않는 것이 원칙이다.

동기화 한 바퀴를 따라가 보기

  • 1) 화면 로드. 공개 호스팅에서 앱(화면)이 열린다. 여기엔 비밀이 없다.
  • 2) 데이터 읽기. 비공개 저장소의 데이터 파일에서 현재 내용을 받아 화면에 뿌린다.
  • 3) 수정. 사용자가 예약을 추가·변경한다. 아직은 화면 안의 변화.
  • 4) 저장(쓰기). 저장 요청을 호스팅 뒤쪽으로 보내면, 그쪽이 env에 숨겨둔 토큰으로 비공개 저장소에 기록한다. 토큰은 화면을 거치지 않는다.
  • 5) 다른 기기. 다른 기기가 앱을 다시 열면 2번에서 최신 데이터를 받아, 방금 바뀐 게 반영돼 보인다.

이 흐름 어디에도 실제 토큰·PIN 값을 화면에 노출하는 단계는 없다. 비밀은 늘 뒤쪽에만 머문다.

이런 데서 자주 막히거나 샌다

  • ① 토큰을 화면 파일에 박기. 브라우저로 누구나 열어본다. → 토큰은 env에, 쓰기는 서버리스 함수에서.
  • ② 데이터를 공개 저장소에 두기. 예약·개인정보가 그대로 노출. → 데이터는 반드시 비공개 저장소.
  • ③ localStorage만 믿기. 그 기기에만 남아 기기 간 공유가 안 된다. → 기기 밖 공용 저장소를 둔다.
  • ④ 파일 더블클릭으로 테스트. file://에서는 불러오기·저장이 조용히 실패한다. → localhost로 확인.
  • ⑤ 공개 노트·문서에 실값 적기. 설명하다 실제 토큰·PIN을 적는 실수. → 언제나 "패턴"으로만 설명.

내 앱 구조로 점검해보기 (5분 연습)

만들(었)던 정적 앱 하나를 떠올려, 아래를 확인해 보세요. 실제 값은 어디에도 적지 말고 "어디에 있나"만 확인합니다.

  • 화면과 데이터가 분리됐나? — 화면은 공개, 데이터는 비공개 저장소에 있는지.
  • 토큰이 화면에 없나? — HTML·JS 안이 아니라 호스팅 env에 있는지.
  • 쓰기가 뒤쪽에서 되나? — 저장 처리가 화면이 아니라 서버리스 함수에서 일어나는지.
  • localhost로 확인했나? — 더블클릭이 아니라 로컬 서버로 불러오기·저장이 실제 되는지.

이 네 가지가 채워지면, 서버 없이도 안전한 기기 간 동기화의 뼈대는 완성입니다.

06 — 점검

배포 전에.

  • ✓화면(공개)과 데이터(비공개)를 분리했는가
  • ✓토큰이 화면 파일에 없고 env에 있는가
  • ✓실제 PIN·토큰 값을 공개 파일에 적지 않았는가
  • ✓localhost로 불러오기·저장을 확인했는가
