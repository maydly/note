규칙을 사람이
매번 못 지킬 때

파일명·폴더·버전 규칙은 만들어도 시간이 지나면 새어나간다. 사람이 매번 검사할 순 없으니, 규칙 기반 스크립트가 주 1회 무인으로 훑어 위반을 찾아 보고하게 만든 이야기. 비용 없이, 자동으로.

자동 감사 규칙 기반 스케줄러 무인 실행

이 노트를 읽고 나면

규칙을 "만드는 것"과 "지켜지게 하는 것"은 다르다.

  • ✓왜 사람의 주의력만으로는 규칙이 안 지켜지는지 안다.
  • ✓무엇을(업무ID·파일명·버전) 자동으로 검사할 수 있는지 안다.
  • ✓왜 AI가 아니라 규칙 스크립트로 하는지(비용·확실성) 안다.
  • ✓감사가 "보고만 하고 고치지 않는" 이유를 안다.
한 줄 요약 — 규칙 기반 스크립트가 매주 한 번, 최근 바뀐 파일만 훑어 규칙 위반을 찾아 보고한다. 사람은 그 보고서를 다음 작업 때 확인해 처리한다. 자동이지만 판단은 사람이.

01 — 왜 자동 감사인가

규칙은 만든 순간부터 새기 시작한다.

파일명·폴더·버전 규칙을 아무리 잘 정해도, 바쁘면 지나친다. 그리고 어긴 걸 아무도 눈치채지 못한 채 쌓인다. 나중에 몰아서 정리하려면 훨씬 큰 일이 된다. 그래서 "어긴 것을 일찍, 자동으로 발견"하는 장치가 필요하다.

개별 작업 사이클(계획→구현→검수)과 별개로 도는 2층 점검이다. 작업 중엔 놓쳐도, 주간 감사가 한 번 더 훑는다.

02 — 무엇을 검사하나

규칙으로 명확히 판정 가능한 것들.

업무ID

업무ID 규칙

산출물이 업무ID 폴더 안에 있는지, ID 형식(대상-분야-번호_업무명)을 지켰는지.

파일명

파일명 규칙

정리·자료 같은 금지 단독 명사, 대상·유형 누락 여부.

버전

버전 표기

버전 파일의 _YYMMDD_HHMM_v버전 형식 준수, 최종본 위치.

03 — 어떻게 도나

주 1회, 최근 바뀐 것만, 무인으로.

1

스케줄러가 깨움

매주 정해진 시각(예: 월요일 새벽)에 OS 스케줄러가 스크립트를 자동 실행한다.

2

최근 변경분만 수집

git 기록으로 최근 7일간 바뀐 파일만 추린다. 전체를 매번 훑지 않아 빠르다.

3

규칙 대조

각 파일을 업무ID·파일명·버전 규칙과 대조해 위반을 찾는다.

4

보고서 누적

결과를 정해진 위치에 HTML 보고서로 쌓는다. 사람이 나중에 확인.

04 — 왜 AI가 아니라 스크립트인가

이 일엔 규칙 기반이 더 맞다.

규칙 스크립트AI 에이전트
비용0 (무료로 무한 반복)호출마다 비용
결과항상 같음(결정적)매번 조금씩 다를 수 있음
적합한 일명확한 규칙 판정맥락·판단이 필요한 일
"형식이 맞나"처럼 규칙으로 딱 떨어지는 검사는 스크립트가 낫다. 비용 없이, 무인으로, 언제나 같은 기준으로 돈다. AI는 판단이 필요한 곳에 아낀다.

05 — 보고만, 고치지 않는다

자동이 사람을 대신하지 않는다.

감사 스크립트는 직접 파일을 옮기거나 이름을 바꾸지 않는다. 위반을 찾아 보고만 하고, 실제 처리는 사람이 다음 작업 사이클에서 확인 후 한다. 자동이 잘못 판단해 멀쩡한 걸 건드리는 사고를 막고, 최종 판단을 사람이 쥔다.

"발견은 자동, 처리는 사람" — 이 분리가 자동화를 안전하게 만든다. 감사는 눈이지 손이 아니다.

왜 "규칙을 만드는 것"과 "지켜지게 하는 것"은 다른가

규칙은 만든 날이 제일 잘 지켜진다. 시간이 지나면 바쁨·피로·깜빡함이 쌓여 조금씩 새어나간다. 사람의 주의력은 한정된 자원이라, 매 파일마다 "이름 규칙 맞나, 버전 표기 맞나"를 검사하는 데 계속 쓸 수 없다. 그래서 규칙은 "의지"로 지키게 두면 반드시 무너진다.

자동 감사는 이걸 정기 건강검진에 비유할 수 있다. 평소엔 몸에 신경 안 써도, 정해진 날 검진이 이상 신호를 잡아준다. 마찬가지로 작업할 땐 규칙을 잊어도, 주 1회 도는 감사가 어긋난 곳을 대신 찾아준다. 사람의 기억력이 아니라 일정(스케줄)에 점검을 맡기는 것이다.

핵심은 "발견"과 "판단"을 나눈 데 있다. 어긋난 것을 찾는 일은 기계가 지치지 않고 잘한다. 반면 "이건 규칙 위반처럼 보여도 사실 예외로 둬야 한다" 같은 판단은 맥락을 아는 사람 몫이다. 자동 감사는 앞의 일만 맡고, 뒤의 일은 그대로 사람에게 남긴다.

주간 감사 하나를 설계한다면 — 순서대로

  • 1) 판정 가능한 규칙만 고른다. "파일명에 정리·자료 단독 사용 금지", "버전 파일은 _YYMMDD_HHMM_v버전 형식" 처럼 기계가 O/X로 딱 나눌 수 있는 것만 대상에 넣는다. "이름이 예쁜가" 같은 건 제외.
  • 2) 검사 범위를 좁힌다. 매번 전체를 훑으면 느리고 무겁다. 변경 이력(git)으로 최근 7일에 바뀐 파일만 추려 그것만 본다.
  • 3) 실행 시각을 못 박는다. "매주 월요일 새벽"처럼 정해두고, OS 스케줄러에 등록해 사람이 켜지 않아도 돌게 한다.
  • 4) 결과를 한곳에 쌓는다. 위반 목록을 정해진 폴더에 보고서로 누적한다. 사람이 다음 작업 때 열어보면 된다.
  • 5) 고치는 권한은 주지 않는다. 스크립트는 "찾아서 적기"까지만. 이름 바꾸기·이동은 사람이 확인 후 직접 한다.

보다시피 감사의 힘은 화려한 기능이 아니라, 범위를 좁히고(2) 권한을 제한한(5) 절제에서 나온다.

자동 감사를 만들 때 이런 데서 자주 어긋난다

  • ① 애매한 것까지 검사하려 함. "제목이 적절한가" 같은 주관적 판단을 규칙에 넣으면, 오탐(멀쩡한 걸 위반이라 함)이 쏟아져 아무도 안 본다. → 딱 떨어지는 것만.
  • ② 매번 전체를 훑음. 파일이 늘수록 점점 느려져, 결국 감사를 꺼버린다. → 최근 변경분만.
  • ③ 스크립트에 수정 권한을 줌. 자동이 잘못 판단해 멀쩡한 파일 이름을 바꾸면 되돌리기 어렵다. → 보고만.
  • ④ 보고서를 아무도 안 보는 곳에 쌓음. 결과가 사람 동선 밖에 있으면 없는 것과 같다. → 작업 중 자연히 보는 위치에.
  • ⑤ 규칙이 바뀌었는데 감사만 옛 규칙. 규칙을 고치면 감사 기준도 함께 갱신해야 한다. → 규칙과 감사를 한 쌍으로 관리.

내 작업에 감사 하나 얹어보기 (5분 연습)

지금 반복해서 어기게 되는 규칙 하나를 떠올려, 아래 네 칸을 채워 보세요. 이게 곧 당신만의 감사 설계도입니다.

  • 어떤 규칙? — 자꾸 새어나가는 규칙 하나를 O/X로 판정 가능하게 다시 적기. (예: "버전 파일에 날짜·버전 표기가 있는가")
  • 검사 범위는? — 전체 말고 "최근 며칠 바뀐 것"으로 좁히기. (예: 최근 7일)
  • 언제 돌리나? — 사람 손 안 가게 정해진 시각 하나. (예: 매주 월요일 아침)
  • 결과를 어디서 보나? — 다음 작업 때 자연히 눈에 띄는 위치 한 곳.

네 칸이 채워지면, 나머지는 "찾기만 하고 고치진 않는" 작은 점검 하나를 얹는 일입니다.

06 — 감사 설계 점검

자동 감사를 만들 때.

  • ✓규칙으로 명확히 판정 가능한 것만 검사 대상인가
  • ✓전체가 아니라 "최근 변경분"만 훑어 가벼운가
  • ✓스크립트가 직접 고치지 않고 보고만 하는가
  • ✓사람이 확인할 위치에 보고서가 쌓이는가
