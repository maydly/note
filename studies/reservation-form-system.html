문의·예약 폼을
시스템으로 만들기

여러 종류의 문의를 한 폼으로 받으면서 뒤로는 데이터가 쌓이게 하는 법. 갈래지는 폼 설계, 필드 종류, Google Apps Script로 스프레드시트에 연결하는 백엔드, 토큰 인증·공개/관리 분리까지 — 예약 시스템을 만들며 배운 것을 정리했다.

폼 설계 Apps Script 스프레드시트 연결 토큰 인증

이 노트를 읽고 나면

"폼 하나"가 아니라 "받는 시스템"을 그린다.

  • ✓여러 종류의 문의를 한 폼에서 갈래로 나누는 설계를 안다.
  • ✓필드 종류와 폼 UX(선택 pill·파일 업로드·동의)를 안다.
  • ✓제출 데이터를 서버 없이 스프레드시트에 쌓는 법을 안다.
  • ✓토큰 인증과 공개/관리 분리로 데이터를 지키는 법을 안다.
한 줄 요약 — 앞(폼)은 "묻는 흐름"을 잘 갈래 짓고, 뒤(백엔드)는 스프레드시트를 데이터함으로 쓰되 토큰으로 잠그고 공개용/관리용을 분리한다.

01 — 갈래지는 폼

한 입구, 여러 길.

문의 종류가 여럿이면(예: 답사·플랫폼 예약·직접 예약·리뷰·상업 문의), 폼을 다섯 개 만드는 대신 하나의 입구에서 먼저 종류를 고르게 하고, 고른 것에 맞는 폼만 보여준다. 사용자는 자기에게 필요한 항목만 보게 되고, 관리자는 한 곳에서 받는다.

1. 선택 화면무엇을 도와줄까
2. 갈래 선택종류 고르기
3. 맞는 폼필요한 항목만
4. 제출데이터 전송
5. 완료 화면안내
"← 다시 선택"으로 언제든 갈래를 바꿀 수 있게 두면, 잘못 고른 사람도 막히지 않는다.

02 — 필드 설계

묻는 방식이 응답의 질을 정한다.

기본

입력 종류

텍스트·이메일·전화·숫자·날짜·URL을 용도에 맞게. 형식이 정해진 건 그 입력 타입으로 받아 실수를 줄인다.

고르기

pill 선택

체크·라디오를 알약(pill) 버튼으로 크게. 손가락으로 누르기 쉽고 보기도 또렷하다.

첨부

파일 업로드

참고 이미지·서류가 필요하면 업로드 필드. 필수/선택을 명확히.

  • 필수 항목은 *로 표시하고, 안내 문구로 "왜 묻는지"를 짧게 붙인다.
  • 세금계산서·동의 같은 조건부 항목은 해당할 때만 펼친다.
  • 제출 버튼은 하나, 눈에 띄게. 누른 뒤엔 완료 화면으로 확실히 넘긴다.

03 — 백엔드: 스프레드시트에 쌓기

서버를 세우지 않고도.

제출된 데이터는 어딘가 쌓여야 한다. 별도 서버·DB 없이 Google Apps Script를 작은 백엔드로 쓰면, 폼 응답을 구글 스프레드시트에 한 줄씩 append 할 수 있다. 시트가 곧 데이터함이자 관리 화면이 된다.

받기

doPost / doGet

Apps Script 웹앱이 요청을 받아 시트에 기록하거나 시트 데이터를 돌려준다.

라우팅

type로 나누기

요청의 type(예: reservations·calendar·config·templates)에 따라 다른 시트를 처리. 한 백엔드가 여러 데이터를 담당.

04 — 토큰 인증 · 공개/관리 분리

데이터를 아무나 못 읽게.

시트에 예약·개인정보가 쌓이므로, 관리용 데이터는 아무나 못 읽어야 한다. 세 가지 장치를 뒀다.

  • ✓토큰 검증 — 스크립트 속성에 비밀 토큰을 두고, 요청의 ?token=이 일치할 때만 데이터를 반환. 없거나 틀리면 거부.
  • ✓공개 앱 / 관리 앱 분리 — 누구나 봐도 되는 공개 데이터(예: 공개 캘린더)와 관리 전용 데이터를 다른 웹앱으로 나눈다.
  • ✓에러 원문 숨김 — 오류가 나도 내부 메시지를 응답에 노출하지 않는다(정보 유출 방지).
  • ✓배포 접근 권한을 "본인만"으로 두면 한 겹 더 안전.
이 원칙은 저장소 보안 노트와 같다 — 비밀(토큰)은 클라이언트에 두지 말고, 실제 토큰 값은 공개 파일·공개 노트에 절대 적지 않는다.

05 — 완료 경험

보낸 뒤가 불안하지 않게.

제출 후 아무 반응이 없으면 사용자는 "된 건가?" 불안해한다. 완료 화면으로 접수됐음을 확실히 알리고, 다음에 무엇이 일어나는지(연락 시점 등)를 한 줄 안내한다. 작은 마무리가 신뢰를 만든다.

왜 "폼 하나"가 아니라 "받는 시스템"으로 봐야 하나

폼은 겉으론 입력칸 몇 개지만, 실제로는 앞·뒤 두 부분으로 이뤄진다. 앞은 사용자가 채우는 화면(폼 UX), 뒤는 그 답이 쌓이고 관리되는 곳(백엔드)이다. 앞만 예쁘게 만들고 뒤를 안 정하면, 답은 들어오는데 어디에 어떻게 쌓이는지 몰라 관리가 무너진다. 그래서 폼은 "한 화면"이 아니라 "받아서 쌓고 지키는 시스템"으로 그려야 한다.

앞쪽의 핵심은 갈래(branch)다. 문의 종류가 여럿이면 폼을 여러 개 만드는 대신, 하나의 입구에서 먼저 종류를 고르게 하고 고른 것에 맞는 항목만 보여준다. 사용자는 자기에게 필요한 것만 보고, 관리자는 한 곳에서 받는다. "한 입구, 여러 길"이다.

뒤쪽의 핵심은 데이터를 어디에 쌓고 어떻게 지키나다. 별도 서버·DB 없이도 작은 스크립트를 백엔드로 써서 스프레드시트에 한 줄씩 쌓을 수 있다. 다만 거기엔 예약·개인정보가 모이므로, 토큰으로 잠그고 공개용/관리용을 분리해야 한다. 앞의 편함과 뒤의 안전을 동시에 잡는 게 폼 시스템의 목표다.

갈래 폼 한 흐름을 따라가 보기

  • 1) 입구에서 종류 선택. "무엇을 도와드릴까요?"로 문의 종류를 먼저 고르게 한다.
  • 2) 맞는 폼만 표시. 고른 종류에 필요한 항목만 펼친다. 나머지는 감춰 사용자가 헤매지 않게.
  • 3) 입력 타입을 맞춘다. 이메일·전화·날짜 등 형식이 정해진 건 그 입력 타입으로 받아 실수를 줄인다. 필수는 *로 표시.
  • 4) 제출 → 뒤쪽에 기록. 답을 백엔드로 보내면 스프레드시트에 한 줄 append. 요청의 type으로 어느 시트에 넣을지 나눈다.
  • 5) 완료 화면. "접수됐고, 다음에 무엇이 일어나는지"를 한 줄로 안내해 불안을 없앤다.

사용자는 1~2번의 "한 입구, 여러 길"만 체감하고, 관리자는 4번에서 한 곳에 모인 데이터를 받는다.

이런 데서 폼 시스템이 무너진다

  • ① 종류마다 폼을 따로. 관리 창구가 흩어져 응답을 놓친다. → 한 입구에서 갈래로 나눈다.
  • ② 되돌아갈 길이 없음. 잘못 고른 사람이 막힌다. → "← 다시 선택"을 항상 둔다.
  • ③ 관리 데이터를 안 잠금. 예약·개인정보를 아무나 읽는다. → 토큰 검증 + 공개용/관리용 분리.
  • ④ 토큰을 클라이언트에 노출. 화면 파일에 실제 토큰을 박으면 그대로 새어나간다. → 비밀은 뒤쪽(스크립트 속성·env)에만, 실값은 공개 파일에 적지 않는다.
  • ⑤ 완료 안내 없음. 제출 후 반응이 없어 "된 건가?" 불안. → 완료 화면으로 접수를 확실히 알린다.

내 문의 폼으로 설계해보기 (5분 연습)

받고 싶은 문의를 떠올려, 아래를 채워 보세요. 실제 토큰·비밀값은 어디에도 적지 않습니다.

  • 갈래는 몇 개? — 문의 종류를 적고, 입구에서 고르게 할 목록으로 정리.
  • 종류별 필수 항목은? — 각 갈래에 꼭 필요한 항목 3~5개와 입력 타입을 적어보기.
  • 어디에 쌓나? — 답이 모일 곳(예: 스프레드시트)과, 요청 type으로 어떻게 나눌지.
  • 어떻게 지키나? — "토큰으로 잠금 + 공개용/관리용 분리"를 내 경우에 맞게 한 문장으로. (실값 없이 방식만)

이 네 칸이 채워지면, 앞은 편하고 뒤는 안전한 폼 시스템의 설계도가 완성됩니다.

06 — 폼 시스템 점검

공개 전에.

  • ✓종류별 갈래가 명확하고, 되돌아갈 수 있는가
  • ✓필수 항목·입력 타입이 알맞은가
  • ✓제출 데이터가 시트에 제대로 쌓이는가
  • ✓관리 데이터가 토큰·분리로 보호되는가
  • ✓완료 화면으로 접수를 알리는가
