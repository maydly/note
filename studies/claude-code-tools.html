Claude Code를
혼자 쓸 때 곁에 둘 도구

비개발 1인 운영, 비용에 민감한 워크플로우(드라이브·HTML·에이전트 사이클 중심)에 맞춰 조사한 보완 도구 정리. 메인은 그대로 두고, 비용 없이 보강하는 쪽으로 골랐다.

  • ✓무료·오픈소스로 보강할 수 있는 보조 코딩 도구를 안다.

  • ✓거를 것·주의할 것을 안다.
기준 시점 — 2026년 6월. 도구의 가격·정책·인수합병은 자주 바뀐다. 실제 도입 직전에 한 번 더 확인할 것.

01 — 메인은 그대로

중심 도구를 자주 바꾸지 않는다. 익숙함도 생산성.

메인은 Claude Code를 유지한다. 구독에 포함돼 추가 비용이 없고, 작업 흐름이 이미 여기 맞춰져 있다. 보완 도구는 "메인을 대체"하려는 게 아니라 "빈 곳을 메우는" 역할로만 고른다.

도구를 바꾸면 익숙함이 초기화된다. 바꾸는 비용(학습·설정·실수)이 얻는 것보다 클 때가 많다. 중심은 고정, 주변만 보강.

02 — 보조 코딩 도구

무료·오픈소스 위주로 부담 없이.

최우선

Cline

오픈소스(무료). 내 API 키를 꽂아 쓰는 방식(BYOK). 보조 코딩의 1순위 후보.

인라인

Copilot Free

무료 등급. 코드를 한 줄씩 자동완성하는 인라인 보조에 유용.

유지. 결과물을 다른 시선으로 교차감사하는 용도.

03 — 연결 확장 (MCP · 커넥터)

AI가 내 자료에 직접 닿게.

MCP·커넥터는 AI를 내 도구·자료에 연결해 주는 다리다. 연동하면 따로 복사해 붙여넣지 않고도 맥락을 가져올 수 있다.

04 — 거를 것 · 주의할 것

고르는 것만큼 안 고르는 것도 중요.

  • !제외 — 인수·정책 변경으로 방향이 달라진 도구는 후보에서 뺀다.
  • !주의 — 연동(OAuth)이 실패하는 사례가 있다. 도입 전 실제 연결을 한 번 테스트한다.
  • !경계 — 도구를 늘리는 것 자체가 목적이 되지 않게. 빈 곳이 있을 때만 더한다.

05 — 고를 때 원칙

이 정리가 맞는 전제.

이 결론은 1인 · 비용민감 · 자료가 드라이브와 HTML 중심인 워크플로우 기준이다. 상황(팀 규모·예산·작업 종류)이 다르면 답도 달라진다. 도구 목록보다 "내 작업의 빈 곳이 어디인가"를 먼저 본다.

📚 더 깊이 — 강의용 상세

위가 요약이라면, 여기는 자세히 배우는 곳. 탭을 눌러 펼쳐 보세요.

왜 "메인은 고정, 주변만 보강"이 답인가

도구를 고를 때 우리는 자꾸 "더 좋은 걸 새로 깔면 나아지지 않을까" 쪽으로 기운다. 하지만 매일 쓰는 메인 도구를 바꾸는 건 손이 익은 부엌을 통째로 새 걸로 바꾸는 것과 같다. 칼이 어디 있는지, 불이 어느 세기인지 몸이 기억하던 게 전부 초기화된다. 그 익숙함을 다시 쌓는 데 드는 시간이 곧 스위칭 비용이다. 새 도구가 조금 더 낫더라도, 이 비용을 넘지 못하면 오히려 손해다.

그래서 이 노트의 전제는 "메인(Claude Code)은 고정하고, 빈 곳만 보조 도구로 메운다"이다. 여기서 중요한 건 순서다. 먼저 "내 작업의 어디가 비었나"를 보고, 그 빈 곳에 맞는 도구를 그다음에 찾는다. 반대로 하면 — 도구 목록을 먼저 훑고 "이거 좋대"부터 깔면 — 필요하지도 않은 걸 늘리게 된다. 도구를 모으는 일 자체가 취미가 되면, 정작 작업은 산만해진다.

보조 도구를 고르는 기준도 단순하게 잡는다 — 비용 부담이 적고(무료·오픈소스), 메인을 대체하지 않으며, 한 가지 빈 곳을 확실히 메우는 것. 1인 · 비용에 민감한 운영에서는 "많이"보다 "꼭 맞게"가 언제나 낫다.

보조 코딩 도구 하나를 실제로 붙인다면 — 순서

새 도구를 고를 때, 목록부터 보지 말고 내 빈 곳부터 본다. 아래 순서로 따라가면 충동구매를 피한다.

  • 1) 빈 곳 확인. 메인으로 하다 답답한 지점을 한 문장으로 적는다. (예: "코드 한 줄씩 자동완성이 아쉽다")
  • 2) 후보 좁히기. 그 빈 곳에 맞는 것만 본다. 인라인 자동완성이 필요하면 Copilot Free 같은 무료 등급부터, 보조 코딩이면 오픈소스인 Cline부터.
  • 3) 비용·연결 확인. 무료인지, 내 API 키(BYOK)를 꽂는 방식인지 확인한다. 연동이 필요하면 실제 연결(OAuth)을 한 번 테스트해 본다.
  • 4) 작게 써보기. 바로 전면 도입하지 않고, 작은 작업 하나에만 붙여 본다. 손에 맞는지 며칠 겪어 본다.
  • 5) 남길지 결정. 그 빈 곳이 실제로 메워졌으면 남기고, 아니면 미련 없이 뺀다. 메인은 이 과정 내내 그대로 둔다.

핵심은 5번까지 가도 메인은 건드리지 않는다는 것. 보조는 어디까지나 곁다리다.

도구를 고를 때 이런 데서 자주 헛돈다

  • ① 메인을 자꾸 갈아탐. "이게 더 좋대"에 흔들려 중심 도구를 바꾸면, 익숙함이 매번 초기화된다. → 메인은 웬만하면 고정.
  • ② 빈 곳 없이 도구부터 깖. 필요를 못 느끼는데 "유명하니까" 설치한다. → 내 작업의 빈 곳을 먼저 적고, 거기 맞는 것만.
  • ③ 비용을 안 따짐. 무료인 줄 알았는데 일정 사용량 넘으면 과금되는 경우가 있다. → 도입 전 요금 조건을 그때그때 확인(이 노트의 가격·정책은 2026.6 기준이라 바뀔 수 있다).
  • ④ 연결을 테스트 안 함. 연동(OAuth)이 실패하는 사례가 있는데, 급할 때 처음 붙이다 막힌다. → 도입 전 실제 연결을 미리 한 번 확인.
  • ⑤ 도구 수집이 목적이 됨. 개수를 늘리는 재미에 빠져 작업이 오히려 산만해진다. → "빈 곳이 있을 때만 더한다"를 규칙으로.

내 도구 구성으로 점검하기 (5분 연습)

지금 쓰는 도구들을 떠올리며 아래 네 칸을 채워 보세요. 이게 곧 "무엇을 남기고 무엇을 뺄지"의 기준이 됩니다.

  • 내 메인은? — 매일 중심으로 쓰는 도구 하나와, 그걸 고정하는 이유를 한 문장으로.
  • 진짜 빈 곳은? — 메인으로 하다 답답한 지점 1~2개를 구체적으로 적어보기.
  • 그 빈 곳에 맞는 후보는? — 각 빈 곳에 무료·오픈소스 후보를 하나씩만. (없으면 "없음"도 답)
  • 지금 뺄 것은? — 깔아만 두고 안 쓰는 도구가 있다면 하나 골라 정리 후보로.

이 네 칸이 채워지면, 도구 목록을 더 뒤질 필요 없이 "내게 필요한 구성"이 이미 보입니다.
