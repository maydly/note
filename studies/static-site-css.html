한 장이 아니라 "사이트"가 되면, 폴더를 어떻게 나누고 CSS를 어떻게 관리하느냐가 전부다. 여러 페이지를 만들며 익힌 정적 사이트 구조와, 화면이 커지든 작아지든 흔들리지 않는 레이아웃의 기본기를 정리했다.

정적 사이트 CSS 레이아웃 구조 설계

이 노트를 읽고 나면

"왜 이렇게 나눴지?"를 나중에도 알 수 있게.

  • ✓공용 CSS와 개별 CSS를 왜 나누는지, 안 나누면 무슨 사고가 나는지 안다.
  • ✓새 페이지를 빠르고 일관되게 찍어내는 템플릿 방식을 안다.
  • ✓레이아웃을 잡는 핵심 CSS 도구 5가지를 쓸 수 있다.
  • ✓사이트를 "가볍게" 유지하는 습관을 안다.

01 — 공용과 개별을 나눈다

랜딩과 하위 페이지는 성격이 다르다. CSS도 나눠야 안전하다.

이 아카이브의 실제 구조다. 첫 화면(랜딩)과 하위 노트는 디자인이 달라서, 스타일 파일을 분리해 두었다. 그래야 한쪽을 고칠 때 다른 쪽이 통째로 깨지는 사고를 막는다.

랜딩 전용

landing.css

첫 화면만 쓰는 스타일. 여기만 고치면 하위 페이지는 안전하다.

노트 공용

note.css

모든 학습 노트가 함께 쓰는 디자인. 한 번 고치면 전 노트에 반영.

개별 파일

studies/

노트 HTML을 모으는 폴더. 템플릿을 복사해 시작한다.

note/
├─ index.html        (랜딩)
├─ landing.css       (랜딩 전용 스타일)
├─ note.css          (노트 공용 스타일)
└─ studies/
   ├─ template.html  (새 노트 원본)
   └─ ...각 노트.html
왜 분리하나 — 모든 페이지가 한 CSS를 공유하면, 랜딩을 예쁘게 바꾸려다 하위 30개 노트가 동시에 깨진다. "영향 범위를 좁히는 것"이 구조 설계의 핵심이다.

02 — 템플릿을 둔다

새 노트를 빈 화면에서 시작하지 않는다.

template.html 하나를 만들어 두고, 새 노트는 그걸 복사해서 제목·내용만 바꾼다. 두 가지가 저절로 따라온다 — 디자인 통일(모두 같은 뼈대)과 제작 속도(매번 처음부터 안 짬).

  • 1template.html을 복사해 studies/새이름.html로 저장
  • 2제목·커버 문구·본문 내용만 교체
  • 3랜딩(index.html)의 해당 섹션에 카드·링크 추가

03 — 흔들리지 않는 레이아웃 도구

이 다섯이면 대부분의 화면이 잡힌다.

중앙·폭

max-width + margin:auto

본문 폭을 적당히 제한하고 가운데 정렬. 큰 화면에서 글이 한없이 늘어나 읽기 힘든 걸 막는다.

.wrap{ max-width:900px; margin:0 auto; }
색·여백 관리

CSS 변수 (:root)

색과 여백을 한 곳에 모아 둔다. 톤을 바꿀 때 변수 한 줄만 고치면 전체가 따라온다.

:root{ --ink:#18212f; }
h1{ color:var(--ink); }
배치

Grid / Flex

카드 나열은 grid, 좌우 정렬은 flex. 이 둘이면 배치는 거의 다 된다.

.cards{ display:grid;
  grid-template-columns:repeat(3,1fr); }
반응형 글자

clamp()

글자 크기를 화면에 따라 자동으로. 최소·기준·최대를 한 줄에.

h1{ font-size:clamp(28px,4.4vw,46px); }
화면 분기

@media

좁은 화면에서 2열을 1열로 바꾸는 식의 반응형. 모바일 대응의 기본.

@media (max-width:720px){
  .cards{ grid-template-columns:1fr; }
}

04 — 밝은 톤을 기본으로

읽는 사람이 편한 색.

이 아카이브는 종이 같은 밝은 배경(오프화이트)을 기본으로 한다. 어두운 색은 강조에만 아주 조금 쓰고, 본문은 읽기 편한 라이트 톤을 유지한다. 포인트 색은 한두 개로 절제하면 차분하면서도 심심하지 않다.

색을 많이 쓰는 것보다, 여백과 정렬이 깔끔함을 만든다. 색은 "여기 봐 달라"는 신호로 아껴 쓴다.

05 — 가볍게 유지하기

정적 사이트의 강점은 속도. 그걸 지키는 습관.

  • ✓이미지는 적정 크기로 줄여서 넣는다. 원본 그대로면 페이지가 무거워진다.
  • ✓이미지를 HTML 안에 통째로 박아넣으면(base64) 파일이 수 MB로 부푼다 — 가능하면 별도 파일로.
  • ✓폰트는 필요한 굵기만 불러온다.
  • ✓외부 스크립트는 꼭 필요한 것만. 정적 사이트의 미덕은 "가벼움"이다.
실제 사례 — 한 학습 노트에 이미지를 HTML 안에 직접 박아 넣었더니 파일 하나가 약 12MB가 됐다. 열 때마다 버벅인다. 이미지는 파일로 분리하는 게 답.

📚 더 깊이 — 강의용 상세

위가 요약이라면, 여기는 자세히 배우는 곳. 탭을 눌러 펼쳐 보세요.

왜 CSS를 "나누는 것"이 실력인가 — 영향 범위라는 개념

페이지가 하나일 땐 CSS를 어떻게 쓰든 티가 안 난다. 문제는 페이지가 늘 때다. 모든 페이지가 하나의 CSS 파일을 함께 쓰면, 편해 보이지만 함정이 있다 — 랜딩(첫 화면)을 예쁘게 바꾸려고 한 줄 고쳤을 뿐인데, 그 스타일을 공유하던 하위 노트 수십 개가 동시에 깨진다. 이걸 "영향 범위가 넓다"고 한다.

그래서 성격이 다른 것끼리 파일을 가른다. 이 아카이브는 랜딩 전용(landing.css)과 노트 공용(note.css)을 나눠 뒀다. 랜딩을 만질 땐 landing.css만 건드리니 하위 노트는 안전하고, 반대로 노트 디자인을 한 번 고치면 note.css 하나로 전 노트에 일괄 반영된다. 핵심은 "고쳤을 때 어디까지 흔들리는가를 미리 좁혀 두는 것" — 이게 구조 설계의 본질이다. 집에서 방마다 두꺼비집(차단기)을 나눠 두는 것과 같다. 한 방을 손봐도 집 전체가 정전되지 않게.

같은 원리로 색·여백 값은 :root의 CSS 변수 한 곳에 모은다. 톤을 바꿀 때 수십 군데를 찾아다니지 않고 변수 한 줄만 고치면 전체가 따라온다. "고칠 곳을 한 곳으로 모으고, 영향 범위를 좁힌다" — 파일 분리도 변수도 결국 같은 생각이다.

새 노트 하나를 만든다면 — 구조를 지키며

  • 1) 템플릿 복사. 빈 화면에서 시작하지 않는다. template.html을 복사해 studies/새이름.html로 저장. 뼈대와 디자인이 저절로 따라온다.
  • 2) 내용만 교체. 제목·커버 문구·본문만 바꾼다. 구조(섹션 뼈대)와 클래스는 그대로 둬야 디자인이 통일된다.
  • 3) 공용 CSS 연결 확인. <link rel="stylesheet" href="../note.css">가 걸려 있는지 본다. 개별 페이지에서 새 CSS 파일을 함부로 만들지 않는다.
  • 4) 색이 필요하면 변수로. 새 색을 직접 박지 말고 :root의 변수(var(--ink) 등)를 쓴다. 톤 일관성이 유지된다.
  • 5) 좁은 화면 점검. 브라우저 폭을 줄여 2열이 1열로 잘 접히는지 본다. @media (max-width:720px) 규칙이 이걸 담당한다.

순서를 지키면 노트가 30개로 늘어도 디자인이 흐트러지지 않고, 한 번의 수정으로 전체를 손볼 수 있다.

구조가 무너지는 지점

  • ① 페이지마다 CSS 복붙. 급하다고 각 HTML에 스타일을 직접 넣으면, 나중에 톤을 바꿀 때 파일마다 다 고쳐야 한다. → 공용은 note.css 한 곳에.
  • ② 색·여백을 값으로 직접 박기. #18212f를 여기저기 하드코딩하면 톤 변경이 지옥이 된다. → :root 변수로 모으고 var()로 참조.
  • ③ 랜딩과 노트 CSS를 한 파일에. 영향 범위가 넓어져 한쪽을 고치면 다른 쪽이 깨진다. → 성격이 다르면 파일을 가른다.
  • ④ 반응형 빠뜨림. PC 화면만 보고 넘기면 폰에서 2열이 터진다. → @media로 좁은 화면 분기를 반드시 넣는다.
  • ⑤ 무거운 이미지·base64. 원본 이미지를 그대로, 혹은 HTML 안에 통째로 박으면 파일이 수 MB로 부푼다. → 적정 크기로 줄이고 별도 파일로 분리한다.

내 사이트로 점검하기 (5분)

지금 만드는(또는 만들려는) 사이트를 떠올리며 아래 네 가지를 답해 보세요.

  • 영향 범위는? — 지금 CSS를 한 줄 고치면 어디까지 바뀌는가? 랜딩과 하위 페이지가 같은 파일을 쓰고 있진 않은가?
  • 고칠 곳은 한 곳인가? — 자주 바꾸는 색·여백이 :root 변수로 모여 있는가, 여기저기 흩어져 있는가?
  • 새 페이지 시작점은? — 복사해서 시작할 template.html이 있는가? 없다면 하나 만들 만한가?
  • 폰과 무게는? — 좁은 화면에서 안 깨지는가, 이미지가 페이지를 무겁게 만들지 않는가?

네 답이 분명하면, 페이지가 늘어도 사이트가 흔들리지 않고 가볍게 유지됩니다.

06 — 새 페이지 점검

올리기 전에 빠르게.

  • ✓공용/개별 CSS를 알맞게 연결했는가
  • ✓모바일(좁은 화면)에서 안 깨지는가
  • ✓밝은 배경·절제된 색을 지켰는가
  • ✓이미지가 무겁지 않은가
