HTML 한 폴더를
공개 주소로 만드는 법

서버도, 비용도 없이 GitHub Pages로 정적 사이트를 띄우는 전체 흐름. 정적 사이트가 뭔지부터 최초 설정·갱신·자주 걸리는 함정까지, 이 노트만 보고 다시 배포할 수 있게 정리했다. 이 아카이브 자체가 이렇게 떠 있다.

GitHub Pages 정적 사이트 무료 호스팅 배포

이 노트를 읽고 나면

한 번 설정하면, 그다음부터는 push 한 번이 곧 배포다.

  • ✓"정적 사이트"가 뭐고 왜 무료·빠른지 안다.
  • ✓GitHub 저장소를 공개 웹주소로 바꾸는 설정을 직접 할 수 있다.
  • ✓수정한 내용을 라이브에 반영하는 법(=push)을 안다.
  • ✓"내 PC에선 되는데 공개 주소에선 깨지는" 단골 함정을 피한다.
한 줄 요약 — GitHub에 올린 HTML/CSS 폴더를, GitHub이 그대로 웹사이트로 공개해 주는 무료 기능이다. 공개 주소는 https://사용자명.github.io/저장소명/ 꼴. 이 아카이브는 bucket0825.github.io/note/.

01 — 정적 사이트 vs 동적 사이트

GitHub Pages는 "정적"에만 맞는다. 그 차이를 먼저.

정적 사이트 (Static)동적 사이트 (Dynamic)
구성HTML·CSS·이미지·JS 파일 그대로서버 프로그램 + 데이터베이스
예시학습 아카이브, 포트폴리오, 안내 페이지로그인·결제·실시간 게시판
비용무료로 가능 (GitHub Pages)서버 운영비 발생
속도매우 빠름 (그냥 파일 전송)요청마다 서버가 계산
학습 노트·안내 페이지처럼 "내용을 보여주기만" 하는 사이트는 정적이면 충분하다. 빠르고, 무료고, 관리가 단순하다.

02 — GitHub Pages가 하는 일

저장소(코드 보관함)를 웹주소로 연결해 준다.

개념

저장소(repository)

GitHub에서 파일을 버전관리하며 보관하는 공간. 평소엔 "코드 창고"다.

기능

Pages

그 창고의 한 폴더를 웹사이트로 공개해 주는 GitHub의 무료 기능. 켜기만 하면 주소가 생긴다.

03 — 최초 설정 (한 번만)

처음 한 번이 끝나면 다신 안 만진다.

1

저장소 만들기

GitHub에서 새 저장소(repository)를 생성한다.

2

파일 올리기

HTML·CSS가 든 폴더를 커밋하고 push 한다.

git add .
git commit -m "first site"
git push origin main
3

Pages 설정 열기

저장소 상단 Settings → Pages로 이동.

4

소스 지정

"Deploy from a branch" 선택 → Branch는 main, 폴더는 / (root).

5

주소 확인

1~2분 뒤 https://사용자명.github.io/저장소명/으로 열린다.

04 — 그 후의 갱신: push = 배포

별도 배포 버튼이 없다. 올리면 바뀐다.

설정이 끝나면 흐름은 아주 단순하다.

1. 수정파일을 고친다
2. commit변경을 저장
3. pushGitHub로 올림
4. 빌드GitHub이 자동 처리
5. 반영1~2분 뒤 라이브
그래서 주의 — "아직 안 보여주고 싶은 것"은 push하지 않는다. push하는 순간 공개된다. 작업 중인 초안은 로컬에만 두거나 별도 브랜치에서.

05 — 자주 걸리는 함정

"내 PC에선 됐는데" 사고의 단골 원인들.

경로

상대경로 쓰기

폴더를 옮기면 링크가 깨진다. 절대경로 대신 ./폴더/파일 같은 상대경로로 쓰고, 구조를 함부로 안 바꾼다.

대소문자

파일명 대소문자

윈도우는 Image.PNG와 image.png를 같게 보지만, 서버는 다르게 본다. 링크와 실제 파일명을 정확히 맞춘다.

캐시

반영이 느릴 때

대개 브라우저 캐시 탓. 강력 새로고침(Ctrl+F5)으로 확인. 그래도 안 되면 1~2분 더 기다린다.

보안

내부 정보 금지

공개 사이트엔 메모·연락처·금액·비밀키를 절대 올리지 않는다. 한 번 push하면 기록에 남는다.

커스텀 도메인(내 주소)을 붙이려면 Pages 설정에서 도메인 입력 + 도메인 업체의 DNS 설정이 추가로 필요하다. 기본 github.io 주소만으로도 충분히 쓸 수 있다.

06 — 배포 전 점검

push 전에 한 번 훑기.

  • ✓모든 링크가 상대경로이고, 실제 파일이 그 자리에 있는가
  • ✓이미지·CSS 파일명 대소문자가 링크와 정확히 같은가
  • ✓공개하면 안 되는 내부 정보가 섞여있지 않은가
  • ✓모바일 화면에서도 깨지지 않는가

07 — 용어 한눈에

이 노트에 나온 단어들.

용어뜻
정적 사이트HTML·CSS·이미지로만 된, 서버 계산이 없는 사이트.
저장소 repositoryGitHub에서 파일을 버전관리하며 보관하는 공간.
GitHub Pages저장소 폴더를 무료 웹사이트로 공개해 주는 기능.
push로컬 커밋을 GitHub로 올리기. Pages에선 곧 배포.
상대경로현재 파일 기준으로 다른 파일 위치를 적는 방식(./, ../).
커스텀 도메인github.io 대신 내 주소를 붙이는 것. DNS 설정 필요.
