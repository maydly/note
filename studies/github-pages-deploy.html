HTML 한 폴더를
공개 주소로 만드는 법

서버도, 비용도 없이 GitHub Pages로 정적 사이트를 띄우는 전체 흐름. 정적 사이트가 뭔지부터 최초 설정·갱신·자주 걸리는 함정까지, 이 노트만 보고 다시 배포할 수 있게 정리했다. 이 아카이브 자체가 이렇게 떠 있다.

GitHub Pages 정적 사이트 무료 호스팅 배포

이 노트를 읽고 나면

한 번 설정하면, 그다음부터는 push 한 번이 곧 배포다.

  • ✓"정적 사이트"가 뭐고 왜 무료·빠른지 안다.
  • ✓GitHub 저장소를 공개 웹주소로 바꾸는 설정을 직접 할 수 있다.
  • ✓수정한 내용을 라이브에 반영하는 법(=push)을 안다.
  • ✓"내 PC에선 되는데 공개 주소에선 깨지는" 단골 함정을 피한다.
한 줄 요약 — GitHub에 올린 HTML/CSS 폴더를, GitHub이 그대로 웹사이트로 공개해 주는 무료 기능이다. 공개 주소는 https://사용자명.github.io/저장소명/ 꼴. 이 아카이브는 bucket0825.github.io/note/.

01 — 정적 사이트 vs 동적 사이트

GitHub Pages는 "정적"에만 맞는다. 그 차이를 먼저.

정적 사이트 (Static)동적 사이트 (Dynamic)
구성HTML·CSS·이미지·JS 파일 그대로서버 프로그램 + 데이터베이스
예시학습 아카이브, 포트폴리오, 안내 페이지로그인·결제·실시간 게시판
비용무료로 가능 (GitHub Pages)서버 운영비 발생
속도매우 빠름 (그냥 파일 전송)요청마다 서버가 계산
학습 노트·안내 페이지처럼 "내용을 보여주기만" 하는 사이트는 정적이면 충분하다. 빠르고, 무료고, 관리가 단순하다.

02 — GitHub Pages가 하는 일

저장소(코드 보관함)를 웹주소로 연결해 준다.

개념

저장소(repository)

GitHub에서 파일을 버전관리하며 보관하는 공간. 평소엔 "코드 창고"다.

기능

Pages

그 창고의 한 폴더를 웹사이트로 공개해 주는 GitHub의 무료 기능. 켜기만 하면 주소가 생긴다.

03 — 최초 설정 (한 번만)

처음 한 번이 끝나면 다신 안 만진다.

1

저장소 만들기

GitHub에서 새 저장소(repository)를 생성한다.

2

파일 올리기

HTML·CSS가 든 폴더를 커밋하고 push 한다.

git add .
git commit -m "first site"
git push origin main
3

Pages 설정 열기

저장소 상단 Settings → Pages로 이동.

4

소스 지정

"Deploy from a branch" 선택 → Branch는 main, 폴더는 / (root).

5

주소 확인

1~2분 뒤 https://사용자명.github.io/저장소명/으로 열린다.

04 — 그 후의 갱신: push = 배포

별도 배포 버튼이 없다. 올리면 바뀐다.

설정이 끝나면 흐름은 아주 단순하다.

1. 수정파일을 고친다
2. commit변경을 저장
3. pushGitHub로 올림
4. 빌드GitHub이 자동 처리
5. 반영1~2분 뒤 라이브
그래서 주의 — "아직 안 보여주고 싶은 것"은 push하지 않는다. push하는 순간 공개된다. 작업 중인 초안은 로컬에만 두거나 별도 브랜치에서.

05 — 자주 걸리는 함정

"내 PC에선 됐는데" 사고의 단골 원인들.

경로

상대경로 쓰기

폴더를 옮기면 링크가 깨진다. 절대경로 대신 ./폴더/파일 같은 상대경로로 쓰고, 구조를 함부로 안 바꾼다.

대소문자

파일명 대소문자

윈도우는 Image.PNG와 image.png를 같게 보지만, 서버는 다르게 본다. 링크와 실제 파일명을 정확히 맞춘다.

캐시

반영이 느릴 때

대개 브라우저 캐시 탓. 강력 새로고침(Ctrl+F5)으로 확인. 그래도 안 되면 1~2분 더 기다린다.

보안

내부 정보 금지

공개 사이트엔 메모·연락처·금액·비밀키를 절대 올리지 않는다. 한 번 push하면 기록에 남는다.

커스텀 도메인(내 주소)을 붙이려면 Pages 설정에서 도메인 입력 + 도메인 업체의 DNS 설정이 추가로 필요하다. 기본 github.io 주소만으로도 충분히 쓸 수 있다.

📚 더 깊이 — 강의용 상세

위가 요약이라면, 여기는 자세히 배우는 곳. 탭을 눌러 펼쳐 보세요.

왜 "정적 사이트"는 무료로, 그것도 GitHub이 대신 띄워 주나

웹사이트를 여는 데는 원래 어딘가 켜져 있는 컴퓨터(서버)가 필요하다. 방문자가 주소를 치면 그 컴퓨터가 파일을 건네주기 때문이다. 그런데 학습 노트나 안내 페이지처럼 미리 만들어 둔 HTML을 그대로 보여주기만 하면 되는 경우엔, 서버가 방문자마다 뭔가 계산할 일이 없다. 그냥 있는 파일을 전달하면 끝이다. 이런 사이트를 정적 사이트라 한다.

바로 여기에 GitHub Pages가 끼어든다. 어차피 GitHub은 전 세계에 파일을 나눠주는 거대한 창고를 이미 운영 중이다. "네 저장소의 이 폴더를 그대로 웹으로 공개해 줄게"가 Pages다. 서버 프로그램도, 데이터베이스도 없으니 GitHub 입장에서 부담이 거의 없어 무료로 내준다. 이미 지어둔 도서관 서가에 내 책 한 권을 꽂아 두는 것과 같다 — 도서관은 이미 돌아가고 있으니 책 한 권 더 얹는 비용은 사실상 0이다.

그래서 흐름이 단순해진다. 내 HTML 폴더를 저장소에 올리고(push) Pages를 한 번 켜면, https://사용자명.github.io/저장소명/ 주소가 생긴다. 이후로는 파일을 고쳐 push하는 것 자체가 곧 배포다. 별도의 "배포 버튼"이 없는 이유가 이것이다.

빈 폴더 하나를 공개 주소로 — 처음부터 끝까지

  • 1) 저장소 만들기. GitHub에서 새 저장소(repository)를 하나 생성한다. 이게 내 파일 창고가 된다.
  • 2) 파일 올리기. index.html과 CSS가 든 폴더를 커밋하고 git push origin main. 첫 화면 파일 이름은 반드시 index.html이어야 자동으로 열린다.
  • 3) Pages 켜기. 저장소의 Settings → Pages로 가서 "Deploy from a branch"를 고르고, Branch는 main, 폴더는 / (root)로 지정.
  • 4) 주소 확인. 1~2분 뒤 https://사용자명.github.io/저장소명/이 열린다. 안 열리면 잠깐 더 기다리거나 강력 새로고침.
  • 5) 이제부터는 push가 배포. 내용을 고치고 commit → push하면 1~2분 뒤 라이브에 반영된다. 3번 설정은 다시 안 만진다.

이 아카이브도 정확히 이 방식으로 떠 있다. 한 번 설정한 뒤로는 "고치고 올리기"만 반복하면 된다.

"내 PC에선 됐는데" 사고의 단골 원인

  • ① 대소문자 어긋남. 윈도우는 Image.PNG와 image.png를 같게 보지만 서버는 다르게 본다. 링크의 대소문자와 실제 파일명을 정확히 맞춘다.
  • ② 절대경로 사용. 내 PC 경로(C:/…)나 고정 절대경로를 쓰면 공개 주소에선 깨진다. ./폴더/파일 같은 상대경로로 쓰고 폴더 구조를 함부로 안 바꾼다.
  • ③ 첫 파일명 실수. 첫 화면 파일이 index.html이 아니면 주소를 열어도 목록만 뜨거나 404가 난다.
  • ④ 캐시를 반영 실패로 오해. push했는데 안 바뀌어 보이면 대개 브라우저 캐시 탓. Ctrl+F5로 강력 새로고침하고, 그래도 안 되면 1~2분 더 기다린다.
  • ⑤ 공개하면 안 될 것을 push. push하는 순간 공개되고 기록에도 남는다. 메모·연락처·금액·비밀키는 절대 올리지 않는다.

내 페이지로 점검하기 (5분)

배포하기 전에, 아래 네 가지를 스스로 확인해 보세요. 대부분의 사고가 여기서 걸러집니다.

  • 첫 파일은? — 공개할 폴더의 첫 화면 파일 이름이 정확히 index.html인가?
  • 링크는? — 이미지·CSS 링크가 모두 상대경로이고, 대소문자까지 실제 파일과 같은가?
  • 공개해도 되나? — 이 폴더 안에 남에게 보이면 안 되는 정보(메모·연락처·키)가 섞여 있지 않은가?
  • 폰에서는? — 좁은 화면에서도 글자·이미지가 안 깨지는지 미리 봤는가?

네 칸이 통과되면, 남은 건 commit과 push뿐입니다.

06 — 배포 전 점검

push 전에 한 번 훑기.

  • ✓모든 링크가 상대경로이고, 실제 파일이 그 자리에 있는가
  • ✓이미지·CSS 파일명 대소문자가 링크와 정확히 같은가
  • ✓공개하면 안 되는 내부 정보가 섞여있지 않은가
  • ✓모바일 화면에서도 깨지지 않는가
