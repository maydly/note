열어보지 않아도
아는
파일 정리법

작업이 쌓일수록 "그 파일 어디 갔지"가 늘어난다. 이름과 폴더, 버전만 잘 잡으면 검색하지 않고도 찾는다. 직접 쓰며 다듬은 규칙.

01 — Naming

이름만 보고 아는 파일

파일명은 "열어봐야 아는" 게 아니라 "보면 아는" 것이어야 한다.

기준

Rule

대상 + 세부내용 + 유형을 모두 이름에 담는다. 정리·자료·리스트 같은 단독 단어는 금지 — 무엇에 대한 건지 알 수 없으니까.

나쁜 예: 리스트.html
좋은 예: 장비_조명비교표_추천리스트.html

날짜·버전 표기

Format

버전 파일은 이름 → 날짜시간 → 버전 순서로 적는다. 날짜를 앞에 두지 않는다.

  • 형식: (파일명)_YYMMDD_HHMM_v버전.확장자
  • 예: 랜딩페이지_260630_0153_v1.1.html
  • 최종본(폴더 밖)에는 날짜·버전을 붙이지 않는다.

02 — Version

덮어쓰지 않는 버전관리

모든 수정은 새 버전으로. 이전으로 돌아갈 길을 항상 남긴다.

원칙

Principle
  • 기존 파일을 덮어쓰지 않는다. 새 버전 파일을 만든다.
  • 최신 최종본은 버전 폴더 바로 밖에 꺼내 둔다.
  • 작은 수정·오타: v1.0 → v1.1
  • 기능 추가·구조 변경: v1.0 → v2.0
  • 이전 버전은 보존한다. 그게 롤백 기준점이 된다.
이름을 바꾸며 발전한 파일이라도, 같은 목적이면 한 버전 폴더에 모아 순서대로 번호를 잇는다. 흩어두면 어느 게 최신인지 알 수 없다.

03 — Tracking

업무 단위로 묶기

파일 하나하나가 아니라, "어느 업무에서 나온 것"인지로 추적한다.

업무ID 구조

Structure

모든 산출물에 업무ID를 붙여 폴더로 묶는다. 형식은 대상-분야-번호_업무명. 그러면 파일 하나만 봐도 "어느 프로젝트의 어떤 업무"인지 바로 안다.

  1. 업무를 시작하면 업무ID 폴더를 만든다.
  2. 그 안에 업무카드(개요·결정·상태)를 둔다.
  3. 계획 → 산출물 → 보고서 → 기록을 정해진 하위 폴더에 쌓는다.
  4. 중단·폐기된 것도 지우지 않고 보존 폴더에 남긴다.
