비밀번호·키를
실수로 올리지 않으려면

공개 저장소에 비밀번호·토큰·개인정보가 한 번 올라가면 지워도 기록에 남는다. 커밋 전에 민감정보를 걸러내고, .gitignore로 막고, 이미 노출됐다면 어떻게 대처하는지 — 실제 점검에서 얻은 원칙을 정리했다.

저장소 보안 커밋 위생 .gitignore 비밀키 관리

이 노트를 읽고 나면

"올리기 전에 뭘 봐야 하는지"가 몸에 밴다.

  • ✓공개 저장소에 절대 올리면 안 되는 것들을 안다.
  • ✓커밋 전에 민감정보를 걸러내는 습관을 안다.
  • ✓.gitignore로 비밀 파일을 아예 추적에서 빼는 법을 안다.
  • ✓이미 노출됐을 때 가장 먼저 할 일을 안다.
한 줄 요약 — git은 "지운 것도 기록에 남는다". 그래서 비밀은 애초에 올리지 않는 것이 유일한 안전책이다. 이미 올라갔다면 그 비밀은 "노출된 것"으로 간주하고 즉시 폐기한다.

01 — 왜 위험한가

클라이언트 파일에 든 비밀은 누구나 본다.

웹페이지 파일(HTML·JS)은 브라우저로 누구나 원본을 열어볼 수 있다. 그 안에 관리자 PIN, 비밀번호 해시, API 토큰이 들어있으면 그대로 노출이다. 실제 점검에서, 공개된 admin.html에 쓰기 권한 토큰이 평문으로 박혀 있던 사례가 있었다 — 그 저장소를 본 사람은 누구나 그 권한을 쓸 수 있는 상태였다.

핵심 — "내 화면에만 보이겠지"는 착각. 클라이언트(브라우저) 파일은 전부 공개라고 가정한다. 비밀은 클라이언트에 두지 않는다.

02 — 절대 커밋 금지 목록

이런 게 섞였는지 커밋 전에 훑는다.

인증

비밀번호 · PIN · 해시

관리자 PIN(0000 같은 실값), 비밀번호, 해시값. 클라이언트 파일이면 특히.

키

토큰 · API 키

GitHub 토큰(ghp_…), 서비스 API 키, 시크릿. .env 파일 전체.

개인정보

PII · 예약 데이터

전화번호·이름·예약 내역 같은 실제 개인정보. 플레이스홀더(010-xxxx)만 안전.

시스템

계정 · 접속 정보

원격접속 계정, 레지스트리 백업, 시스템 로그·해시 백업.

03 — .gitignore로 아예 막기

사람 기억이 아니라 규칙으로 차단.

.gitignore에 비밀 파일 패턴을 적어두면, git이 그 파일들을 애초에 추적하지 않는다. 실수로 git add .를 해도 안 올라간다. 사람의 주의력에 기대지 말고 규칙으로 막는다.

# .gitignore 예시
*.env
.env*
admin*.html        # 관리자 페이지(실값 보유)
**/worklog/        # 비번 기록
*.reg              # 레지스트리 백업
rdp*.txt           # 원격접속 계정
점검에서 실제로 한 것 — 저장소마다 .gitignore를 보강해 비밀 파일 패턴을 넓히고, README의 PIN 실값을 ****로 정리했다. "안전 파일만" 골라 커밋.

04 — 이미 노출됐다면

지우는 게 아니라, 폐기가 먼저.

1

즉시 폐기(revoke)

노출된 토큰·키는 "이미 새어나간 것". 파일에서 지우는 것보다 발급처에서 폐기가 먼저다. GitHub 토큰이면 즉시 revoke.

2

새로 발급

폐기한 뒤 새 키를 발급받아, 이번엔 클라이언트가 아닌 안전한 곳에 둔다.

3

공개 범위 확인

레포가 public인지 private인지, 어디까지 퍼졌는지 확인.

4

파일 정리 + 재발 방지

파일에서 제거하고 .gitignore에 추가. (기록에서 완전 삭제하려면 별도 이력 정리가 필요.)

순서가 중요하다 — 파일만 지우고 안심하면 안 된다. 노출된 비밀은 이미 유효하지 않게 만드는 것(폐기)이 진짜 대처다.

05 — 근본 구조: 비밀을 분리

애초에 클라이언트에 비밀이 없게.

가장 좋은 건 비밀이 클라이언트 파일에 들어갈 일 자체를 없애는 것이다. 인증·키가 필요한 처리는 브라우저가 아니라 뒤쪽(서버·서버리스 함수)에서 하고, 클라이언트엔 결과만 받는다. 관리자 페이지의 PIN도 개발용은 플레이스홀더로 치환한 버전을 따로 둔다.

위험한 방식안전한 방식
HTML/JS 안에 토큰·PIN 직접 박기비밀은 서버/환경변수, 클라이언트엔 없음
실값을 그대로 커밋플레이스홀더 버전만 커밋

📚 더 깊이 — 강의용 상세

위가 요약이라면, 여기는 자세히 배우는 곳. 탭을 눌러 펼쳐 보세요.

왜 "지운 것도 남고", "클라이언트는 다 공개"인가

git은 파일의 변경 이력을 통째로 쌓아두는 도구다. 지금 파일에서 비밀을 지워도, 그 비밀이 들어 있던 과거 커밋은 그대로 남아 있다. 마치 일기장에서 한 줄을 지우개로 지워도 앞장에 그대로 적혀 있는 것과 같다. 그래서 "올렸다가 지우면 되겠지"가 통하지 않는다. 한 번 올라간 비밀은 이미 새어나간 것으로 봐야 한다.

또 하나 착각하기 쉬운 게 "내 화면에만 보이겠지"다. 웹페이지를 이루는 HTML·JS는 브라우저가 받아서 실행하는 파일이라, 누구나 그 원본을 열어볼 수 있다. 화면에 안 보이게 숨겨도 파일 안에 값이 있으면 그대로 노출이다. 그래서 비밀(PIN·토큰·키)은 클라이언트(브라우저) 쪽에 두는 순간 공개라고 가정해야 한다.

이 두 성질을 합치면 결론은 하나다 — 비밀은 애초에 올리지 않는 것이 유일한 안전책이고, 그걸 사람 기억이 아니라 규칙(.gitignore)으로 막는 것이다. 기술이 아니라 습관과 구조의 문제라, 코딩을 잘 몰라도 지킬 수 있다.

커밋 전 30초 점검 흐름

  • 1) 무엇을 올리는지 먼저 본다. git add .로 몰아 올리기 전에, 바뀐 파일 목록을 훑는다. 낯선 파일(설정·백업·로그)이 섞였는지 확인.
  • 2) 위험 단어를 훑는다. 바뀐 내용에서 password, token, key, secret, 전화번호·이름 같은 패턴이 있는지 눈으로 스캔.
  • 3) 비밀 파일은 추적에서 뺀다. 설정·환경 파일(.env류), 관리자용 실값 페이지, 기록 폴더는 .gitignore에 패턴으로 넣어 아예 안 올라가게 한다.
  • 4) 실값을 플레이스홀더로 바꾼다. 안내 문서에 예시값이 필요하면 실제 값 대신 xxxx 같은 가짜값으로.
  • 5) 그다음에 커밋. "안전한 파일만 남았다"가 확인되면 그때 올린다.

여기 어떤 단계에도 실제 비밀값을 적을 일은 없다. 흐름 자체가 "값을 빼는 방향"으로 흐른다.

이런 데서 자주 새어나간다

  • ① 관리자 페이지에 PIN 직접 박기. 클라이언트 파일이라 그대로 공개된다. → 인증은 뒤쪽(서버)에서 하고, 개발용은 플레이스홀더 버전을 따로 둔다.
  • ② git add .로 다 몰아 올리기. 옆에 있던 설정·백업 파일까지 딸려 올라간다. → 무엇이 올라가는지 목록을 먼저 본다.
  • ③ 지웠으니 안전하다는 착각. 과거 커밋에 남아 있다. → 노출된 비밀은 "지우기"가 아니라 "폐기(재발급)"가 먼저.
  • ④ 안내문에 실제 값 예시. README에 실 PIN·토큰을 예시로 적는 실수. → 예시는 항상 가짜값(****)으로.
  • ⑤ .gitignore를 나중으로 미룸. 규칙이 없으면 결국 사람 기억에 기댄다. → 프로젝트 시작할 때 비밀 패턴부터 넣는다.

내 저장소로 점검해보기 (5분 연습)

지금 올리려는(또는 이미 올린) 저장소 하나를 떠올려, 아래를 하나씩 확인해 보세요. 실제 값은 어디에도 적지 말고, "있다/없다"만 체크하면 됩니다.

  • 비밀이 클라이언트에 있나? — HTML·JS 안에 PIN·토큰·키가 박혀 있는지. 있으면 뒤쪽으로 옮길 대상.
  • .gitignore가 막고 있나? — 환경·설정·기록 파일 패턴이 들어 있는지 한 줄씩 확인.
  • 안내문에 실값이 있나? — README·문서의 예시값을 플레이스홀더로 바꿀 곳 찾기.
  • 노출됐다면 폐기 순서를 아는가? — "발급처에서 폐기 → 새로 발급 → 안전한 곳에 보관"을 한 문장으로 적어보기.

이 네 가지가 "괜찮다"로 채워지면, 커밋 전 보안의 큰 구멍은 대부분 막힌 셈입니다.

06 — 커밋 전 보안 점검

push 전에 반드시.

  • ✓비밀번호·PIN·해시가 파일에 없는가
  • ✓토큰·API 키·.env가 섞이지 않았는가
  • ✓실제 개인정보(전화·이름·예약)가 없는가
  • ✓비밀 파일이 .gitignore로 막혀 있는가
  • ✓(노출 시) 폐기부터 했는가
