Development · maydly

비밀번호·키를
실수로 올리지 않으려면

공개 저장소에 비밀번호·토큰·개인정보가 한 번 올라가면 지워도 기록에 남는다. 커밋 전에 민감정보를 걸러내고, .gitignore로 막고, 이미 노출됐다면 어떻게 대처하는지 — 실제 점검에서 얻은 원칙을 정리했다.

저장소 보안 커밋 위생 .gitignore 비밀키 관리

이 노트를 읽고 나면

"올리기 전에 뭘 봐야 하는지"가 몸에 밴다.

  • ✓공개 저장소에 절대 올리면 안 되는 것들을 안다.
  • ✓커밋 전에 민감정보를 걸러내는 습관을 안다.
  • ✓.gitignore로 비밀 파일을 아예 추적에서 빼는 법을 안다.
  • ✓이미 노출됐을 때 가장 먼저 할 일을 안다.
한 줄 요약 — git은 "지운 것도 기록에 남는다". 그래서 비밀은 애초에 올리지 않는 것이 유일한 안전책이다. 이미 올라갔다면 그 비밀은 "노출된 것"으로 간주하고 즉시 폐기한다.

01 — 왜 위험한가

클라이언트 파일에 든 비밀은 누구나 본다.

웹페이지 파일(HTML·JS)은 브라우저로 누구나 원본을 열어볼 수 있다. 그 안에 관리자 PIN, 비밀번호 해시, API 토큰이 들어있으면 그대로 노출이다. 실제 점검에서, 공개된 admin.html에 쓰기 권한 토큰이 평문으로 박혀 있던 사례가 있었다 — 그 저장소를 본 사람은 누구나 그 권한을 쓸 수 있는 상태였다.

핵심 — "내 화면에만 보이겠지"는 착각. 클라이언트(브라우저) 파일은 전부 공개라고 가정한다. 비밀은 클라이언트에 두지 않는다.

02 — 절대 커밋 금지 목록

이런 게 섞였는지 커밋 전에 훑는다.

인증

비밀번호 · PIN · 해시

관리자 PIN(0825 같은 실값), 비밀번호, 해시값. 클라이언트 파일이면 특히.

키

토큰 · API 키

GitHub 토큰(ghp_…), 서비스 API 키, 시크릿. .env 파일 전체.

개인정보

PII · 예약 데이터

전화번호·이름·예약 내역 같은 실제 개인정보. 플레이스홀더(010-xxxx)만 안전.

시스템

계정 · 접속 정보

원격접속 계정, 레지스트리 백업, 시스템 로그·해시 백업.

03 — .gitignore로 아예 막기

사람 기억이 아니라 규칙으로 차단.

.gitignore에 비밀 파일 패턴을 적어두면, git이 그 파일들을 애초에 추적하지 않는다. 실수로 git add .를 해도 안 올라간다. 사람의 주의력에 기대지 말고 규칙으로 막는다.

# .gitignore 예시
*.env
.env*
admin*.html        # 관리자 페이지(실값 보유)
**/worklog/        # 비번 기록
*.reg              # 레지스트리 백업
rdp*.txt           # 원격접속 계정
점검에서 실제로 한 것 — 저장소마다 .gitignore를 보강해 비밀 파일 패턴을 넓히고, README의 PIN 실값을 ****로 정리했다. "안전 파일만" 골라 커밋.

04 — 이미 노출됐다면

지우는 게 아니라, 폐기가 먼저.

1

즉시 폐기(revoke)

노출된 토큰·키는 "이미 새어나간 것". 파일에서 지우는 것보다 발급처에서 폐기가 먼저다. GitHub 토큰이면 즉시 revoke.

2

새로 발급

폐기한 뒤 새 키를 발급받아, 이번엔 클라이언트가 아닌 안전한 곳에 둔다.

3

공개 범위 확인

레포가 public인지 private인지, 어디까지 퍼졌는지 확인.

4

파일 정리 + 재발 방지

파일에서 제거하고 .gitignore에 추가. (기록에서 완전 삭제하려면 별도 이력 정리가 필요.)

순서가 중요하다 — 파일만 지우고 안심하면 안 된다. 노출된 비밀은 이미 유효하지 않게 만드는 것(폐기)이 진짜 대처다.

05 — 근본 구조: 비밀을 분리

애초에 클라이언트에 비밀이 없게.

가장 좋은 건 비밀이 클라이언트 파일에 들어갈 일 자체를 없애는 것이다. 인증·키가 필요한 처리는 브라우저가 아니라 뒤쪽(서버·서버리스 함수)에서 하고, 클라이언트엔 결과만 받는다. 관리자 페이지의 PIN도 개발용은 플레이스홀더로 치환한 버전을 따로 둔다.

위험한 방식안전한 방식
HTML/JS 안에 토큰·PIN 직접 박기비밀은 서버/환경변수, 클라이언트엔 없음
실값을 그대로 커밋플레이스홀더 버전만 커밋

06 — 커밋 전 보안 점검

push 전에 반드시.

  • ✓비밀번호·PIN·해시가 파일에 없는가
  • ✓토큰·API 키·.env가 섞이지 않았는가
  • ✓실제 개인정보(전화·이름·예약)가 없는가
  • ✓비밀 파일이 .gitignore로 막혀 있는가
  • ✓(노출 시) 폐기부터 했는가

07 — 용어 한눈에

이 노트에 나온 단어들.

토큰 / API 키서비스에 접근할 권한을 담은 비밀 문자열.
PII개인 식별 정보(이름·전화·예약 등).
