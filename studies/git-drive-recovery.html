git가 갑자기
“커밋이 없다”고
말할 때

Google Drive 위에 둔 git 저장소가 동기화 충돌로 깨졌을 때, 작업을 잃지 않고 되살리는 순서. 실제로 한 번 겪고 복구한 기록을 정리한 노트.

01 — Why

왜 이런 일이 생기나

git의 .git 폴더와 클라우드 동기화가 같은 파일을 동시에 만지면서 생기는 충돌.

원인 한 줄

Cause

git은 커밋을 마칠 때 .git 폴더 안의 작은 파일들(refs, index, HEAD)을 아주 빠르게 만들고 이름을 바꾼다. 이 폴더가 Google Drive 동기화 대상이면, Drive가 그 찰나에 끼어들어 파일을 덮거나 잠가버린다. 그 결과 git이 "어디까지 커밋했는지"를 기록한 표식이 사라진다.

핵심 증상: git status가 “커밋 없음(No commits yet)”이라고 한다. 분명 한참 작업했는데도. 그리고 .git/refs/heads/ 안에 main 대신 main.lock만 남아 있다.

왜 위험한가

Risk

이 상태에서 그냥 다시 커밋하면, git은 처음부터 새로 만드는 것으로 착각한다. 잘못하면 빈 내용이나 옛 내용이 원격(GitHub)으로 올라가 그동안의 작업이 통째로 덮일 수 있다. 그래서 "복구가 먼저, 커밋은 나중"이 철칙이다.

02 — Recover

잃지 않고 되살리는 순서

원격(GitHub)을 항상 진실 기준으로 두고, 로컬 작업물을 보존하며 표식만 복원한다.

복구 7단계

Steps
  1. 멈춘다. 이 상태에서는 절대 커밋하지 않는다.
  2. 작업 파일 내용이 살아있는지부터 확인한다 (파일 크기·내용 눈으로 확인).
  3. 스테일 락을 지운다 — .git 안의 *.lock 파일 삭제.
  4. 사라진 표식을 복원한다 — 마지막 커밋 값으로 refs/heads/main 되살리기.
  5. GitHub 원격의 진짜 현재 상태를 직접 확인한다 (git ls-remote).
  6. 로컬과 원격이 갈라졌으면, 먼저 백업 브랜치를 만든 뒤 합친다(rebase).
  7. 의도한 파일만 바뀐 게 맞는지 확인하고, 그제서야 커밋·push.
무엇을 하든 백업 브랜치를 먼저 만들어 두면 마음이 편하다. git branch backup-날짜 main 한 줄이면, 이후 무슨 일이 나도 원래 자리로 돌아올 수 있다.

갈라진 역사 합치기

Merge

로컬에서 작업한 사이 다른 곳(클라우드 작업·다른 기기)에서도 커밋이 쌓여 두 갈래로 나뉘는 일이 있다. 이때는 한쪽을 버리지 말고, 로컬의 새 커밋을 원격 위에 다시 얹는다(rebase). 같은 파일을 양쪽이 고쳤으면 충돌이 나는데, 보통 둘 다 살리는 게 답일 때가 많다 — 서로 다른 기능을 더한 것뿐이라.

03 — Prevent

다시 안 겪으려면

근본 해결과, 당장 지킬 습관.

남길 교훈

Lesson
  • git 저장소는 가능하면 클라우드 동기화 폴더 밖에 두는 게 근본 해결.
  • GitHub(원격)이 항상 진실 기준이다. 로컬이 이상하면 원격과 대조부터.
  • 이상 징후(커밋 없음 표시, .lock 파일)가 보이면 커밋을 멈추고 점검.
  • 중요한 작업 전후로 push 해두면, 그 자체가 복구 기준점이 된다.
