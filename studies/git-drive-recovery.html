git가 갑자기
“커밋이 없다”고 말할 때

Google Drive 위에 둔 git 저장소가 동기화 충돌로 깨졌을 때, 작업을 단 하나도 잃지 않고 되살리는 전체 과정. 개념 → 증상 → 원인 → 복구 → 분기 합치기 → 예방 순서로, 나중에 다시 겪어도 이 노트만 보고 처리할 수 있게 정리했다.

Git Google Drive 트러블슈팅 실제 복구 기록

이 노트를 읽고 나면

실제로 한 번 터졌고, 아래 순서대로 전부 복구한 기록이다.

  • ✓git이 작업을 "어떻게" 저장하는지 감을 잡는다 — 그래야 뭐가 깨졌는지 이해된다.
  • ✓git status가 "커밋 없음"이라고 거짓말하는 상황의 정체를 안다.
  • ✓작업물을 잃지 않고 되살리는 명령을 순서대로 따라 칠 수 있다.
  • ✓로컬과 GitHub이 따로 흘러갔을 때(분기) 둘 다 살려서 합칠 수 있다.
  • ✓다시는 안 겪도록 근본 예방책을 안다.
한 줄 요약 — Drive가 git의 내부 폴더(.git)를 건드려 "마지막 커밋이 어디"라는 표식이 사라진 상태다. 표식만 되살리면 작업은 그대로 돌아온다. 당황해서 다시 커밋하지 않는 것이 제일 중요하다.

01 — 먼저, git은 작업을 어떻게 저장하나

이 구조를 알면 "뭐가 깨졌는지"가 한눈에 보인다.

git은 저장할 때마다 그 순간의 전체 상태를 커밋(commit)이라는 덩어리로 만든다. 각 커밋에는 9b7b488… 같은 고유 번호(해시)가 붙고, "내 직전 커밋은 누구"라는 연결고리를 갖는다. 그래서 커밋들은 사슬처럼 이어진다.

개념

커밋 (commit)

저장 시점의 스냅샷. 고유 해시로 식별되고, 직전 커밋을 가리켜 사슬을 이룬다. 되돌아갈 수 있는 "기준점".

개념

브랜치 / ref

main 같은 브랜치는 사실 "최신 커밋 해시 하나를 적어둔 작은 종이쪽지"다. 이 쪽지를 ref라 하고 .git/refs/heads/main 파일에 들어있다.

개념

HEAD

"지금 내가 보고 있는 위치". 보통 main을 가리킨다. .git/HEAD 파일에 들어있다.

포인트 — 실제 작업 내용(커밋 덩어리)과, "최신이 누구인지 적은 쪽지(ref)"는 별개의 파일이다. 이번 사고는 작업 내용은 멀쩡한데 쪽지만 사라진 경우였다. 그래서 쪽지만 다시 써주면 복구된다.

02 — 증상: git이 “커밋이 없다”고 한다

한참 작업했는데도 처음 상태처럼 보인다.

git status를 쳤더니 이렇게 나왔다 —

$ git status
On branch main

No commits yet            <-- ?! 분명 수십 번 커밋했는데

Changes to be committed:
  new file:   index.html
  new file:   styles.css
  ... (모든 파일이 "새 파일"로 보임)

그리고 git log는 이렇게 거부했다 —

$ git log
fatal: your current branch 'main' does not have any commits yet

결정적 단서는 .git 안에 있었다. 원래 main 파일이 있어야 할 자리에 main.lock만 남아 있었다 —

$ ls .git/refs/heads/
main.lock      <-- main은 사라지고 .lock만 남음
해석 — "최신 커밋이 누구"라고 적은 쪽지(refs/heads/main)가 사라졌다. 그래서 git은 "이 브랜치엔 커밋이 하나도 없네"라고 착각한다. 커밋 덩어리 자체는 .git 안에 그대로 살아있다.

03 — 원인: Drive가 .git을 건드렸다

클라우드 동기화와 git이 같은 파일을 동시에 만질 때 생긴다.

git은 커밋을 마칠 때 .git/refs/heads/main을 안전하게 바꾸려고, 먼저 main.lock에 새 값을 쓰고 → 그 파일을 main으로 이름만 바꾸는 방식을 쓴다(원자적 교체). 그런데 이 폴더가 Google Drive 동기화 대상이면, Drive가 그 짧은 순간에 끼어들어 파일을 잠그거나 동기화해버린다. 그러면 "이름 바꾸기"가 중간에 끊겨 main은 사라지고 main.lock만 덩그러니 남는다.

왜 위험한가 — 이 상태에서 당황해서 그냥 git add → git commit을 하면, git은 "처음 만드는 저장소"로 알고 빈 역사를 새로 쓴다. 그걸 push하면 GitHub의 멀쩡한 작업이 빈 내용으로 덮일 수 있다. 그래서 무조건 "복구 먼저, 커밋 나중".

04 — 복구 순서 (실제로 친 명령)

원격(GitHub)을 진실 기준으로, 로컬 작업물을 보존하며 표식만 되살린다.

1

멈춘다 — 커밋 금지

이 상태에서 커밋하면 역사가 꼬인다. 손을 떼고 상태부터 파악한다.

2

작업물이 살아있는지 확인

파일 크기·내용을 직접 본다. 비어있지 않고 내용이 그대로면 안심. (이번엔 전부 멀쩡했다.)

3

스테일 락 삭제

중간에 끊겨 남은 잠금 파일을 지운다.

find .git -name "*.lock" -delete
4

사라진 쪽지(ref) 복원

다행히 main.lock 안에 마지막 커밋 해시가 들어있었다. 그 값으로 main을 되살린다.

git update-ref refs/heads/main <마지막커밋해시>

이 순간 git log가 정상으로 돌아온다.

5

GitHub의 진짜 현재 상태 확인

로컬에 캐시된 원격 정보는 오래됐을 수 있다. 실제 GitHub에 물어본다.

git ls-remote origin main
6

로컬 ↔ 원격 관계 파악

둘이 얼마나 벌어졌는지 본다.

git fetch origin
git rev-list --count origin/main..main   # 로컬만 가진 커밋 수
git rev-list --count main..origin/main   # 원격만 가진 커밋 수
7

백업 → 합치기 → 확인 후 push

분기됐다면 다음 섹션으로. 아니면 백업 후 push.

git branch backup-260630 main   # 안전망
git push origin main
가장 중요한 한 줄 — 무엇을 하기 전이든 git branch backup-날짜 main으로 백업 브랜치를 먼저 만든다. 이후 어떤 실수를 해도 이 백업으로 원래 자리에 돌아올 수 있다. 마음의 안전벨트.

05 — 갈라진 역사 합치기 (실제 사례)

로컬과 GitHub이 같은 지점에서 각자 다른 길로 간 경우.

이번 실제 상황이 딱 이랬다. 6월 19일 한 지점(공통 조상)에서 갈라져서 —

GitHub 쪽

41개 커밋

다른 환경에서 작업한 장비 페이지 정비. 로컬은 이걸 한 번도 못 받았다.

로컬 쪽

2개 커밋

이 PC에서만 한 작업(홈 네비 + 교육페이지 v4.0). GitHub엔 없다.

한쪽을 버리면 안 된다. 양쪽 다 진짜 작업이니까. 해법은 rebase — 로컬의 2개 커밋을 GitHub의 41개 커밋 "위에 다시 얹는" 것. 그러면 41 + 2 = 전부 보존된다.

git branch backup-260630 main      # 1) 먼저 백업
git rebase origin/main             # 2) 로컬 커밋을 원격 위로 재배치
rebase가 뭔가 — "내 커밋들을 떼어내서, 최신 기준점 위에 하나씩 다시 쌓기". 합치는 또 다른 방법인 merge와 달리 역사가 한 줄로 깔끔하게 남는다. 대신 다시 쌓는 과정에서 충돌이 날 수 있다(다음 섹션).

06 — 충돌이 나면

양쪽이 같은 파일의 같은 부분을 고쳤을 때.

rebase 중 같은 파일을 양쪽이 건드렸으면 git이 멈추고 충돌을 표시한다. 파일을 열면 이런 표식이 들어있다 —

<<<<<<< HEAD
(GitHub 쪽이 넣은 내용)
=======
(내가 넣은 내용)
>>>>>>> 내 커밋

이번엔 양쪽이 서로 다른 CSS 규칙을 더한 것뿐이라, 답은 간단했다 — 둘 다 남기고 표식 세 줄만 지웠다. 그리고 —

git add <충돌해결한파일>
git rebase --continue
충돌은 "둘 중 하나 고르기"가 아닐 때가 많다. 대개 둘 다 필요한 변경이다. 당황하지 말고 내용을 읽어보면 보통 합칠 수 있다. 정 모르겠으면 git rebase --abort로 시작 전으로 안전하게 되돌린다.

07 — 다시 안 겪으려면

근본 해결과, 당장 지킬 습관.

  • ✓근본 해결: git 저장소는 가능하면 클라우드 동기화 폴더 밖(로컬 디스크)에 둔다. .git이 Drive와 안 부딪히게.
  • ✓진실 기준은 GitHub. 로컬이 이상하면 항상 원격과 대조부터 한다.
  • ✓징후 감지: "커밋 없음" 표시나 .lock 파일이 보이면 즉시 커밋을 멈추고 점검한다.
  • ✓자주 push: 중요한 작업 전후로 push 해두면, 그 자체가 복구 기준점이 된다.
  • ✓백업 습관: 위험한 작업 전엔 git branch backup-날짜 한 줄.

08 — 용어 한눈에

이 노트에 나온 단어들 정리.

용어뜻
커밋 commit저장 시점의 스냅샷. 고유 해시로 식별.
ref"최신 커밋이 누구"인지 적어둔 쪽지 파일. refs/heads/main.
HEAD지금 보고 있는 위치. 보통 현재 브랜치를 가리킴.
origin원격 저장소(GitHub)의 기본 별칭.
.lock 파일git이 파일을 안전하게 바꾸는 동안 거는 임시 잠금. 끊기면 "스테일 락"으로 남음.
fetch원격의 최신 정보를 가져오기(작업트리는 안 바꿈).
rebase내 커밋들을 다른 기준점 위에 다시 쌓기. 역사를 한 줄로.
충돌 conflict양쪽이 같은 부분을 다르게 고쳐 git이 자동 합치기를 못 하는 상태.
분기 diverge공통 조상에서 로컬과 원격이 각자 다른 커밋을 쌓아 갈라진 상태.
